210/* IBM compiler versions are a bit messy. There are actually two products:
211 the C product, and the C++ product. The C++ compiler is always packaged
212 with the latest version of the C compiler. Version numbers do not always
213 match. This little table (I'm not sure it's accurate) should be helpful:
214
215 C++ product C product
216
217 C Set 3.1 C Compiler 3.0
218 ... ...
219 C++ Compiler 3.6.6 C Compiler 4.3
220 ... ...
221 Visual Age C++ 4.0 ...
222 ... ...
223 Visual Age C++ 5.0 C Compiler 5.0
224 ... ...
225 Visual Age C++ 6.0 C Compiler 6.0
226
227 Now:
228 __xlC__ is the version of the C compiler in hexadecimal notation
229 is only an approximation of the C++ compiler version
230 __IBMCPP__ is the version of the C++ compiler in decimal notation
231 but it is not defined on older compilers like C Set 3.1 */
232#elif defined(__xlC__)
233# define Q_CC_XLC
234# if __xlC__ < 0x400
235# error "Compiler not supported"
236# elif __xlC__ >= 0x0600
237# define Q_TYPEOF(expr) __typeof__(expr)
238# define Q_PACKED __attribute__((__packed__))
239# endif

1066 *
1067 * 1) make them fully inline; and
1068 * 2) guard them using the necessary feature-testing macros.
1069 *
1070 * This decouples the C++ version used to build Qt with the one used by
1071 * end-user applications; Qt and the application can either choose any C++

1074 * A problem arises on MSVC for member functions of exported classes. Client
