12/*!
13 \class QSystemSemaphore
14 \inmodule QtCore
15 \since 4.4
16
17 \brief The QSystemSemaphore class provides a general counting system semaphore.
18
19 A semaphore is a generalization of a mutex. While a mutex can be
20 locked only once, a semaphore can be acquired multiple times.
21 Typically, a semaphore is used to protect a certain number of
22 identical resources.
23
24 Like its lighter counterpart QSemaphore, a QSystemSemaphore can be
25 accessed from multiple \l {QThread} {threads}. Unlike QSemaphore, a
26 QSystemSemaphore can also be accessed from multiple \l {QProcess}
27 {processes}. This means QSystemSemaphore is a much heavier class, so
28 if your application doesn't need to access your semaphores across
29 multiple processes, you will probably want to use QSemaphore.
30
31 Semaphores support two fundamental operations, acquire() and release():
32
33 acquire() tries to acquire one resource. If there isn't a resource
34 available, the call blocks until a resource becomes available. Then
35 the resource is acquired and the call returns.
36
37 release() releases one resource so it can be acquired by another
38 process. The function can also be called with a parameter n > 1,
39 which releases n resources.
40
41 A system semaphore is created with a string key that other processes
42 can use to use the same semaphore.
43
44 Example: Create a system semaphore
45 \snippet code/src_corelib_kernel_qsystemsemaphore.cpp 0
46
47 A typical application of system semaphores is for controlling access
48 to a circular buffer shared by a producer process and a consumer
49 processes.
50
51 \section1 Platform-Specific Behavior
52
53 When using this class, be aware of the following platform
54 differences:
55
56 \b{Windows:} QSystemSemaphore does not own its underlying system
57 semaphore. Windows owns it. This means that when all instances of
58 QSystemSemaphore for a particular key have been destroyed, either by
59 having their destructors called, or because one or more processes
60 crash, Windows removes the underlying system semaphore.
61
62 \b{Unix:}
63
64 \list
65 \li QSystemSemaphore owns the underlying system semaphore
66 in Unix systems. This means that the last process having an instance of
67 QSystemSemaphore for a particular key must remove the underlying
68 system semaphore in its destructor. If the last process crashes
69 without running the QSystemSemaphore destructor, Unix does not
70 automatically remove the underlying system semaphore, and the
71 semaphore survives the crash. A subsequent process that constructs a
72 QSystemSemaphore with the same key will then be given the existing
73 system semaphore. In that case, if the QSystemSemaphore constructor
74 has specified its \l {QSystemSemaphore::AccessMode} {access mode} as
75 \l {QSystemSemaphore::} {Open}, its initial resource count will not
76 be reset to the one provided but remain set to the value it received
77 in the crashed process. To protect against this, the first process
78 to create a semaphore for a particular key (usually a server), must
79 pass its \l {QSystemSemaphore::AccessMode} {access mode} as \l
80 {QSystemSemaphore::} {Create}, which will force Unix to reset the
81 resource count in the underlying system semaphore.
82
83 \li When a process using QSystemSemaphore terminates for
84 any reason, Unix automatically reverses the effect of all acquire
85 operations that were not released. Thus if the process acquires a
86 resource and then exits without releasing it, Unix will release that
87 resource.
88
89 \endlist
90
91 \b{Apple platforms:} Sandboxed applications (including apps
92 shipped through the Apple App Store) require the key to
93 be in the form \c {<application group identifier>/<custom identifier>},
94 as documented \l {https://developer.apple.com/library/archive/documentation/Security/Conceptual/AppSandboxDesignGuide/AppSandboxInDepth/AppSandboxInDepth.html#//apple_ref/doc/uid/TP40011183-CH3-SW24}
95 {here} and \l {https://developer.apple.com/documentation/bundleresources/entitlements/com_apple_security_application-groups}
96 {here}, and the key length is limited to 30 characters.
97
98 \sa QSharedMemory, QSemaphore
99 */
100
101/*!
102 Requests a system semaphore for the specified \a key. The parameters
103 \a initialValue and \a mode are used according to the following
104 rules, which are system dependent.
105
106 In Unix, if the \a mode is \l {QSystemSemaphore::} {Open} and the
107 system already has a semaphore identified by \a key, that semaphore
108 is used, and the semaphore's resource count is not changed, i.e., \a
109 initialValue is ignored. But if the system does not already have a
110 semaphore identified by \a key, it creates a new semaphore for that
111 key and sets its resource count to \a initialValue.
112
113 In Unix, if the \a mode is \l {QSystemSemaphore::} {Create} and the
114 system already has a semaphore identified by \a key, that semaphore
115 is used, and its resource count is set to \a initialValue. If the
116 system does not already have a semaphore identified by \a key, it
117 creates a new semaphore for that key and sets its resource count to
118 \a initialValue.
119
120 In Windows, \a mode is ignored, and the system always tries to
121 create a semaphore for the specified \a key. If the system does not
122 already have a semaphore identified as \a key, it creates the
123 semaphore and sets its resource count to \a initialValue. But if the
124 system already has a semaphore identified as \a key it uses that
125 semaphore and ignores \a initialValue.
126
127 The \l {QSystemSemaphore::AccessMode} {mode} parameter is only used
128 in Unix systems to handle the case where a semaphore survives a
129 process crash. In that case, the next process to allocate a
130 semaphore with the same \a key will get the semaphore that survived
131 the crash, and unless \a mode is \l {QSystemSemaphore::} {Create},
132 the resource count will not be reset to \a initialValue but will
133 retain the initial value it had been given by the crashed process.
134
135 \sa acquire(), key()
136 */
137QSystemSemaphore::QSystemSemaphore(const QString &key, int initialValue, AccessMode mode)
138 : d(new QSystemSemaphorePrivate)
139{
140 setKey(key, initialValue, mode);
141}
142
143/*!
144 The destructor destroys the QSystemSemaphore object, but the
145 underlying system semaphore is not removed from the system unless
146 this instance of QSystemSemaphore is the last one existing for that
147 system semaphore.
148
149 Two important side effects of the destructor depend on the system.
150 In Windows, if acquire() has been called for this semaphore but not
151 release(), release() will not be called by the destructor, nor will
152 the resource be released when the process exits normally. This would
153 be a program bug which could be the cause of a deadlock in another
154 process trying to acquire the same resource. In Unix, acquired
155 resources that are not released before the destructor is called are
156 automatically released when the process exits.
157*/
158QSystemSemaphore::~QSystemSemaphore()
159{
160 d->cleanHandle();
161}
162
163/*!
164 \enum QSystemSemaphore::AccessMode
165
166 This enum is used by the constructor and setKey(). Its purpose is to
167 enable handling the problem in Unix implementations of semaphores
168 that survive a crash. In Unix, when a semaphore survives a crash, we
169 need a way to force it to reset its resource count, when the system
170 reuses the semaphore. In Windows, where semaphores can't survive a
171 crash, this enum has no effect.
172
173 \value Open If the semaphore already exists, its initial resource
174 count is not reset. If the semaphore does not already exist, it is
175 created and its initial resource count set.
176
177 \value Create QSystemSemaphore takes ownership of the semaphore and
178 sets its resource count to the requested value, regardless of
179 whether the semaphore already exists by having survived a crash.
180 This value should be passed to the constructor, when the first
181 semaphore for a particular key is constructed and you know that if
182 the semaphore already exists it could only be because of a crash. In
183 Windows, where a semaphore can't survive a crash, Create and Open
184 have the same behavior.
185*/
186
187/*!
188 This function works the same as the constructor. It reconstructs
189 this QSystemSemaphore object. If the new \a key is different from
190 the old key, calling this function is like calling the destructor of
191 the semaphore with the old key, then calling the constructor to
192 create a new semaphore with the new \a key. The \a initialValue and
193 \a mode parameters are as defined for the constructor.
194
195 \sa QSystemSemaphore(), key()
196 */
197void QSystemSemaphore::setKey(const QString &key, int initialValue, AccessMode mode)
198{
199 if (key == d->key && mode == Open)
200 return;
201 d->clearError();
202#if !defined(Q_OS_WIN) && !defined(QT_POSIX_IPC)
203 // optimization to not destroy/create the file & semaphore
204 if (key == d->key && mode == Create && d->createdSemaphore && d->createdFile) {
205 d->initialValue = initialValue;
206 d->unix_key = -1;
207 d->handle(mode);
208 return;
209 }
210#endif
211 d->cleanHandle();
212 d->key = key;
213 d->initialValue = initialValue;
214 // cache the file name so it doesn't have to be generated all the time.
215 d->fileName = d->makeKeyFileName();
216 d->handle(mode);
217}

222
223 \sa setKey()

225QString QSystemSemaphore::key() const

230/*!
231 Acquires one of the resources guarded by this semaphore, if there is
232 one available, and returns \c true. If all the resources guarded by this
233 semaphore have already been acquired, the call blocks until one of
234 them is released by another process or thread having a semaphore
235 with the same key.
236
237 If false is returned, a system error has occurred. Call error()
238 to get a value of QSystemSemaphore::SystemSemaphoreError that
239 indicates which error occurred.
240
241 \sa release()
242 */
243bool QSystemSemaphore::acquire()
244{
245 return d->modifySemaphore(count: -1);
246}
247
248/*!
249 Releases \a n resources guarded by the semaphore. Returns \c true
250 unless there is a system error.
251
252 Example: Create a system semaphore having five resources; acquire
253 them all and then release them all.
254
255 \snippet code/src_corelib_kernel_qsystemsemaphore.cpp 1
256
257 This function can also "create" resources. For example, immediately
258 following the sequence of statements above, suppose we add the
259 statement:
260
261 \snippet code/src_corelib_kernel_qsystemsemaphore.cpp 2
262
263 Ten new resources are now guarded by the semaphore, in addition to
264 the five that already existed. You would not normally use this
265 function to create more resources.
266
267 \sa acquire()
268 */
269bool QSystemSemaphore::release(int n)
270{
271 if (n == 0)
272 return true;
273 if (n < 0) {
274 qWarning(msg: "QSystemSemaphore::release: n is negative.");
275 return false;
276 }
277 return d->modifySemaphore(count: n);
278}
