17/*!
18 \class QTimer
19 \inmodule QtCore
20 \brief The QTimer class provides repetitive and single-shot timers.
21
22 \ingroup events
23
24
25 The QTimer class provides a high-level programming interface for
26 timers. To use it, create a QTimer, connect its timeout() signal
27 to the appropriate slots, and call start(). From then on, it will
28 emit the timeout() signal at constant intervals.
29
30 Example for a one second (1000 millisecond) timer (from the
31 \l{widgets/analogclock}{Analog Clock} example):
32
33 \snippet ../widgets/widgets/analogclock/analogclock.cpp 4
34 \snippet ../widgets/widgets/analogclock/analogclock.cpp 5
35 \snippet ../widgets/widgets/analogclock/analogclock.cpp 6
36
37 From then on, the \c update() slot is called every second.
38
39 You can set a timer to time out only once by calling
40 setSingleShot(true). You can also use the static
41 QTimer::singleShot() function to call a slot after a specified
42 interval:
43
44 \snippet timers/timers.cpp 3
45
46 In multithreaded applications, you can use QTimer in any thread
47 that has an event loop. To start an event loop from a non-GUI
48 thread, use QThread::exec(). Qt uses the timer's
49 \l{QObject::thread()}{thread affinity} to determine which thread
50 will emit the \l{QTimer::}{timeout()} signal. Because of this, you
51 must start and stop the timer in its thread; it is not possible to
52 start a timer from another thread.
53
54 As a special case, a QTimer with a timeout of 0 will time out as soon as
55 possible, though the ordering between zero timers and other sources of
56 events is unspecified. Zero timers can be used to do some work while still
57 providing a snappy user interface:
58
59 \snippet timers/timers.cpp 4
60 \snippet timers/timers.cpp 5
61 \snippet timers/timers.cpp 6
62
63 From then on, \c processOneThing() will be called repeatedly. It
64 should be written in such a way that it always returns quickly
65 (typically after processing one data item) so that Qt can deliver
66 events to the user interface and stop the timer as soon as it has done all
67 its work. This is the traditional way of implementing heavy work
68 in GUI applications, but as multithreading is nowadays becoming available on
69 more and more platforms, we expect that zero-millisecond
70 QTimer objects will gradually be replaced by \l{QThread}s.
71
72 \section1 Accuracy and Timer Resolution
73
74 The accuracy of timers depends on the underlying operating system
75 and hardware. Most platforms support a resolution of 1 millisecond,
76 though the accuracy of the timer will not equal this resolution
77 in many real-world situations.
78
79 The accuracy also depends on the \l{Qt::TimerType}{timer type}. For
80 Qt::PreciseTimer, QTimer will try to keep the accuracy at 1 millisecond.
81 Precise timers will also never time out earlier than expected.
82
83 For Qt::CoarseTimer and Qt::VeryCoarseTimer types, QTimer may wake up
84 earlier than expected, within the margins for those types: 5% of the
85 interval for Qt::CoarseTimer and 500 ms for Qt::VeryCoarseTimer.
86
87 All timer types may time out later than expected if the system is busy or
88 unable to provide the requested accuracy. In such a case of timeout
89 overrun, Qt will emit timeout() only once, even if multiple timeouts have
90 expired, and then will resume the original interval.
91
92 \section1 Alternatives to QTimer
93
94 An alternative to using QTimer is to call QObject::startTimer()
95 for your object and reimplement the QObject::timerEvent() event
96 handler in your class (which must inherit QObject). The
97 disadvantage is that timerEvent() does not support such
98 high-level features as single-shot timers or signals.
99
100 Another alternative is QBasicTimer. It is typically less
101 cumbersome than using QObject::startTimer()
102 directly. See \l{Timers} for an overview of all three approaches.
103
104 Some operating systems limit the number of timers that may be
105 used; Qt tries to work around these limitations.
106
107 \sa QBasicTimer, QTimerEvent, QObject::timerEvent(), Timers,
108 {Analog Clock}, {Tetrix Example}
109*/

176/*! \overload start()
177
178 Starts or restarts the timer with the timeout specified in \l interval.
179
180 If the timer is already running, it will be
181 \l{QTimer::stop()}{stopped} and restarted.
182
183 If \l singleShot is true, the timer will be activated only once.
184*/
185void QTimer::start()
186{
187 Q_D(QTimer);
188 if (d->id != QTimerPrivate::INV_TIMER) // stop running timer
189 stop();
190 d->id = QObject::startTimer(interval: d->inter, timerType: d->type);
191 d->isActiveData.notify();
192}

194/*!
195 Starts or restarts the timer with a timeout interval of \a msec
196 milliseconds.
197
198 If the timer is already running, it will be
199 \l{QTimer::stop()}{stopped} and restarted.
200
201 If \l singleShot is true, the timer will be activated only once.
202
203 \note Keeping the event loop busy with a zero-timer is bound to
204 cause trouble and highly erratic behavior of the UI.
205*/
206void QTimer::start(int msec)
207{
208 Q_D(QTimer);
209 const bool intervalChanged = msec != d->inter;
210 d->inter.setValue(msec);
211 start();
212 if (intervalChanged)
213 d->inter.notify();
214}

273QSingleShotTimer::QSingleShotTimer(int msec, Qt::TimerType timerType, const QObject *r, QtPrivate::QSlotObjectBase *slotObj)
274 : QObject(QAbstractEventDispatcher::instance()), hasValidReceiver(r), receiver(r), slotObj(slotObj)

293void QSingleShotTimer::timerEvent(QTimerEvent *)
294{
295 // need to kill the timer _before_ we emit timeout() in case the
296 // slot connected to timeout calls processEvents()
297 if (timerId > 0)
298 killTimer(id: timerId);
299 timerId = -1;
300
301 if (slotObj) {
302 // If the receiver was destroyed, skip this part
303 if (Q_LIKELY(!receiver.isNull() || !hasValidReceiver)) {
304 // We allocate only the return type - we previously checked the function had
305 // no arguments.
306 void *args[1] = { nullptr };
307 slotObj->call(r: const_cast<QObject*>(receiver.data()), a: args);
308 }
309 } else {
310 emit timeout();
311 }
312
313 // we would like to use delete later here, but it feels like a
314 // waste to post a new event to handle this event, so we just unset the flag
315 // and explicitly delete...
316 qDeleteInEventHandler(o: this);
317}

319/*!
320 \internal
321
322 Implementation of the template version of singleShot
323
324 \a msec is the timer interval
325 \a timerType is the timer type
326 \a receiver is the receiver object, can be null. In such a case, it will be the same
327 as the final sender class.
328 \a slot a pointer only used when using Qt::UniqueConnection
329 \a slotObj the slot object
330 */
331void QTimer::singleShotImpl(int msec, Qt::TimerType timerType,
332 const QObject *receiver,
333 QtPrivate::QSlotObjectBase *slotObj)
334{
335 if (msec == 0) {
336 bool deleteReceiver = false;
337 // Optimize: set a receiver context when none is given, such that we can use
338 // QMetaObject::invokeMethod which is more efficient than going through a timer.
339 // We need a QObject living in the current thread. But the QThread itself lives
340 // in a different thread - with the exception of the main QThread which lives in
341 // itself. And QThread::currentThread() is among the few QObjects we know that will
342 // most certainly be there. Note that one can actually call singleShot before the
343 // QApplication is created!
344 if (!receiver && QThread::currentThread() == QCoreApplicationPrivate::mainThread()) {
345 // reuse main thread as context object
346 receiver = QThread::currentThread();
347 } else if (!receiver) {
348 // Create a receiver context object on-demand. According to the benchmarks,
349 // this is still more efficient than going through a timer.
350 receiver = new QObject;
351 deleteReceiver = true;
352 }
353
354 QMetaObject::invokeMethodImpl(object: const_cast<QObject *>(receiver), slot: slotObj,
355 type: Qt::QueuedConnection, ret: nullptr);
356
357 if (deleteReceiver)
358 const_cast<QObject *>(receiver)->deleteLater();
359 return;
360 }
361
362 new QSingleShotTimer(msec, timerType, receiver, slotObj);
363}

365/*!
366 \reentrant
367 This static function calls a slot after a given time interval.
368
369 It is very convenient to use this function because you do not need
370 to bother with a \l{QObject::timerEvent()}{timerEvent} or
371 create a local QTimer object.
372
373 Example:
374 \snippet code/src_corelib_kernel_qtimer.cpp 0
375
376 This sample program automatically terminates after 10 minutes
377 (600,000 milliseconds).
378
379 The \a receiver is the receiving object and the \a member is the
380 slot. The time interval is \a msec milliseconds.
381
382 \sa start()
383*/
384
385void QTimer::singleShot(int msec, const QObject *receiver, const char *member)
386{
387 // coarse timers are worst in their first firing
388 // so we prefer a high precision timer for something that happens only once
389 // unless the timeout is too big, in which case we go for coarse anyway
390 singleShot(msec, timerType: msec >= 2000 ? Qt::CoarseTimer : Qt::PreciseTimer, receiver, member);
391}

393/*! \overload
394 \reentrant
395 This static function calls a slot after a given time interval.
396
397 It is very convenient to use this function because you do not need
398 to bother with a \l{QObject::timerEvent()}{timerEvent} or
399 create a local QTimer object.
400
401 The \a receiver is the receiving object and the \a member is the slot. The
402 time interval is \a msec milliseconds. The \a timerType affects the
403 accuracy of the timer.
404
405 \sa start()
406*/
407void QTimer::singleShot(int msec, Qt::TimerType timerType, const QObject *receiver, const char *member)
408{
409 if (Q_UNLIKELY(msec < 0)) {
410 qWarning(msg: "QTimer::singleShot: Timers cannot have negative timeouts");
411 return;
412 }
413 if (receiver && member) {
414 if (msec == 0) {
415 // special code shortpath for 0-timers
416 const char* bracketPosition = strchr(s: member, c: '(');
417 if (!bracketPosition || !(member[0] >= '0' && member[0] <= '2')) {
418 qWarning(msg: "QTimer::singleShot: Invalid slot specification");
419 return;
420 }
421 QByteArray methodName(member+1, bracketPosition - 1 - member); // extract method name
422 QMetaObject::invokeMethod(obj: const_cast<QObject *>(receiver), member: methodName.constData(), c: Qt::QueuedConnection);
423 return;
424 }
425 (void) new QSingleShotTimer(msec, timerType, receiver, member);
426 }
427}

693 \property QTimer::interval
694 \brief the timeout interval in milliseconds
695
696 The default value for this property is 0. A QTimer with a timeout
697 interval of 0 will time out as soon as all the events in the window
698 system's event queue have been processed.
699
700 Setting the interval of an active timer changes its timerId().
701
702 \sa singleShot
703*/
704void QTimer::setInterval(int msec)
705{
706 Q_D(QTimer);
707 const bool intervalChanged = msec != d->inter;
708 d->inter.setValue(msec);
709 if (d->id != QTimerPrivate::INV_TIMER) { // create new timer
710 QObject::killTimer(id: d->id); // restart timer
711 d->id = QObject::startTimer(interval: msec, timerType: d->type);
712 // No need to call markDirty() for d->isActiveData here,
713 // as timer state actually does not change
714 }
715 if (intervalChanged)
716 d->inter.notify();
717}
