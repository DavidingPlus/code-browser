5/*!
6 \class QUrl
7 \inmodule QtCore
8
9 \brief The QUrl class provides a convenient interface for working
10 with URLs.
11
12 \reentrant
13 \ingroup io
14 \ingroup network
15 \ingroup shared
16
17 It can parse and construct URLs in both encoded and unencoded
18 form. QUrl also has support for internationalized domain names
19 (IDNs).
20
21 The most common way to use QUrl is to initialize it via the constructor by
22 passing a QString containing a full URL. QUrl objects can also be created
23 from a QByteArray containing a full URL using QUrl::fromEncoded(), or
24 heuristically from incomplete URLs using QUrl::fromUserInput(). The URL
25 representation can be obtained from a QUrl using either QUrl::toString() or
26 QUrl::toEncoded().
27
28 URLs can be represented in two forms: encoded or unencoded. The
29 unencoded representation is suitable for showing to users, but
30 the encoded representation is typically what you would send to
31 a web server. For example, the unencoded URL
32 "http://bÃ¼hler.example.com/List of applicants.xml"
33 would be sent to the server as
34 "http://xn--bhler-kva.example.com/List%20of%20applicants.xml".
35
36 A URL can also be constructed piece by piece by calling
37 setScheme(), setUserName(), setPassword(), setHost(), setPort(),
38 setPath(), setQuery() and setFragment(). Some convenience
39 functions are also available: setAuthority() sets the user name,
40 password, host and port. setUserInfo() sets the user name and
41 password at once.
42
43 Call isValid() to check if the URL is valid. This can be done at any point
44 during the constructing of a URL. If isValid() returns \c false, you should
45 clear() the URL before proceeding, or start over by parsing a new URL with
46 setUrl().
47
48 Constructing a query is particularly convenient through the use of the \l
49 QUrlQuery class and its methods QUrlQuery::setQueryItems(),
50 QUrlQuery::addQueryItem() and QUrlQuery::removeQueryItem(). Use
51 QUrlQuery::setQueryDelimiters() to customize the delimiters used for
52 generating the query string.
53
54 For the convenience of generating encoded URL strings or query
55 strings, there are two static functions called
56 fromPercentEncoding() and toPercentEncoding() which deal with
57 percent encoding and decoding of QString objects.
58
59 fromLocalFile() constructs a QUrl by parsing a local
60 file path. toLocalFile() converts a URL to a local file path.
61
62 The human readable representation of the URL is fetched with
63 toString(). This representation is appropriate for displaying a
64 URL to a user in unencoded form. The encoded form however, as
65 returned by toEncoded(), is for internal use, passing to web
66 servers, mail clients and so on. Both forms are technically correct
67 and represent the same URL unambiguously -- in fact, passing either
68 form to QUrl's constructor or to setUrl() will yield the same QUrl
69 object.
70
71 QUrl conforms to the URI specification from
72 \l{RFC 3986} (Uniform Resource Identifier: Generic Syntax), and includes
73 scheme extensions from \l{RFC 1738} (Uniform Resource Locators). Case
74 folding rules in QUrl conform to \l{RFC 3491} (Nameprep: A Stringprep
75 Profile for Internationalized Domain Names (IDN)). It is also compatible with the
76 \l{http://freedesktop.org/wiki/Specifications/file-uri-spec/}{file URI specification}
77 from freedesktop.org, provided that the locale encodes file names using
78 UTF-8 (required by IDN).

96 \section2 Error checking
97
98 QUrl is capable of detecting many errors in URLs while parsing it or when
99 components of the URL are set with individual setter methods (like
100 setScheme(), setHost() or setPath()). If the parsing or setter function is
101 successful, any previously recorded error conditions will be discarded.
102
103 By default, QUrl setter methods operate in QUrl::TolerantMode, which means
104 they accept some common mistakes and mis-representation of data. An
105 alternate method of parsing is QUrl::StrictMode, which applies further
106 checks. See QUrl::ParsingMode for a description of the difference of the
107 parsing modes.
108
109 QUrl only checks for conformance with the URL specification. It does not
110 try to verify that high-level protocol URLs are in the format they are
111 expected to be by handlers elsewhere. For example, the following URIs are
112 all considered valid by QUrl, even if they do not make sense when used:
113
114 \list
115 \li "http:/filename.html"
116 \li "mailto://example.com"
117 \endlist
118
119 When the parser encounters an error, it signals the event by making
120 isValid() return false and toString() / toEncoded() return an empty string.
121 If it is necessary to show the user the reason why the URL failed to parse,
122 the error condition can be obtained from QUrl by calling errorString().
123 Note that this message is highly technical and may not make sense to
124 end-users.
125
126 QUrl is capable of recording only one error condition. If more than one
127 error is found, it is undefined which error is reported.
128
129 \section2 Character Conversions
130

132 dealing with URLs and strings:
133
134 \list
135 \li When creating a QString to contain a URL from a QByteArray or a
136 char*, always use QString::fromUtf8().
137 \endlist

140/*!
141 \enum QUrl::ParsingMode
142
143 The parsing mode controls the way QUrl parses strings.
144
145 \value TolerantMode QUrl will try to correct some common errors in URLs.
146 This mode is useful for parsing URLs coming from sources
147 not known to be strictly standards-conforming.
148
149 \value StrictMode Only valid URLs are accepted. This mode is useful for
150 general URL validation.
151
152 \value DecodedMode QUrl will interpret the URL component in the fully-decoded form,
153 where percent characters stand for themselves, not as the beginning
154 of a percent-encoded sequence. This mode is only valid for the
155 setters setting components of a URL; it is not permitted in
156 the QUrl constructor, in fromEncoded() or in setUrl().
157 For more information on this mode, see the documentation for
158 \l {QUrl::ComponentFormattingOption}{QUrl::FullyDecoded}.
159
160 In TolerantMode, the parser has the following behaviour:
161
162 \list
163
164 \li Spaces and "%20": unencoded space characters will be accepted and will
165 be treated as equivalent to "%20".
166
167 \li Single "%" characters: Any occurrences of a percent character "%" not
168 followed by exactly two hexadecimal characters (e.g., "13% coverage.html")
169 will be replaced by "%25". Note that one lone "%" character will trigger
170 the correction mode for all percent characters.
171
172 \li Reserved and unreserved characters: An encoded URL should only
173 contain a few characters as literals; all other characters should
174 be percent-encoded. In TolerantMode, these characters will be
175 accepted if they are found in the URL:
176 space / double-quote / "<" / ">" / "\" /
177 "^" / "`" / "{" / "|" / "}"
178 Those same characters can be decoded again by passing QUrl::DecodeReserved
179 to toString() or toEncoded(). In the getters of individual components,
180 those characters are often returned in decoded form.
181
182 \endlist
183
184 When in StrictMode, if a parsing error is found, isValid() will return \c
185 false and errorString() will return a message describing the error.
186 If more than one error is detected, it is undefined which error gets
187 reported.
188
189 Note that TolerantMode is not usually enough for parsing user input, which
190 often contains more errors and expectations than the parser can deal with.
191 When dealing with data coming directly from the user -- as opposed to data
192 coming from data-transfer sources, such as other programs -- it is
193 recommended to use fromUserInput().
194
195 \sa fromUserInput(), setUrl(), toString(), toEncoded(), QUrl::FormattingOptions
196*/

234/*!
235 \enum QUrl::ComponentFormattingOption
236 \since 5.0
237
238 The component formatting options define how the components of an URL will
239 be formatted when written out as text. They can be combined with the
240 options from QUrl::FormattingOptions when used in toString() and
241 toEncoded().
242
243 \value PrettyDecoded The component is returned in a "pretty form", with
244 most percent-encoded characters decoded. The exact
245 behavior of PrettyDecoded varies from component to
246 component and may also change from Qt release to Qt
247 release. This is the default.
248
249 \value EncodeSpaces Leave space characters in their encoded form ("%20").
250
251 \value EncodeUnicode Leave non-US-ASCII characters encoded in their UTF-8
252 percent-encoded form (e.g., "%C3%A9" for the U+00E9
253 codepoint, LATIN SMALL LETTER E WITH ACUTE).
254
255 \value EncodeDelimiters Leave certain delimiters in their encoded form, as
256 would appear in the URL when the full URL is
257 represented as text. The delimiters are affected
258 by this option change from component to component.
259 This flag has no effect in toString() or toEncoded().
260
261 \value EncodeReserved Leave US-ASCII characters not permitted in the URL by
262 the specification in their encoded form. This is the
263 default on toString() and toEncoded().
264
265 \value DecodeReserved Decode the US-ASCII characters that the URL specification
266 does not allow to appear in the URL. This is the
267 default on the getters of individual components.
268
269 \value FullyEncoded Leave all characters in their properly-encoded form,
270 as this component would appear as part of a URL. When
271 used with toString(), this produces a fully-compliant
272 URL in QString form, exactly equal to the result of
273 toEncoded()
274
275 \value FullyDecoded Attempt to decode as much as possible. For individual
276 components of the URL, this decodes every percent
277 encoding sequence, including control characters (U+0000
278 to U+001F) and UTF-8 sequences found in percent-encoded form.
279 Use of this mode may cause data loss, see below for more information.
280
281 The values of EncodeReserved and DecodeReserved should not be used together
282 in one call. The behavior is undefined if that happens. They are provided
283 as separate values because the behavior of the "pretty mode" with regards
284 to reserved characters is different on certain components and specially on
285 the full URL.
286
287 \section2 Full decoding
288
289 The FullyDecoded mode is similar to the behavior of the functions returning
290 QString in Qt 4.x, in that every character represents itself and never has
291 any special meaning. This is true even for the percent character ('%'),
292 which should be interpreted to mean a literal percent, not the beginning of
293 a percent-encoded sequence. The same actual character, in all other
294 decoding modes, is represented by the sequence "%25".
295
296 Whenever re-applying data obtained with QUrl::FullyDecoded into a QUrl,
297 care must be taken to use the QUrl::DecodedMode parameter to the setters
298 (like setPath() and setUserName()). Failure to do so may cause
299 re-interpretation of the percent character ('%') as the beginning of a
300 percent-encoded sequence.
301
302 This mode is quite useful when portions of a URL are used in a non-URL
303 context. For example, to extract the username, password or file paths in an
304 FTP client application, the FullyDecoded mode should be used.
305
306 This mode should be used with care, since there are two conditions that
307 cannot be reliably represented in the returned QString. They are:
308
309 \list
310 \li \b{Non-UTF-8 sequences:} URLs may contain sequences of
311 percent-encoded characters that do not form valid UTF-8 sequences. Since
312 URLs need to be decoded using UTF-8, any decoder failure will result in
313 the QString containing one or more replacement characters where the
314 sequence existed.
315
316 \li \b{Encoded delimiters:} URLs are also allowed to make a distinction
317 between a delimiter found in its literal form and its equivalent in
318 percent-encoded form. This is most commonly found in the query, but is
319 permitted in most parts of the URL.
320 \endlist
321
322 The following example illustrates the problem:
323
324 \snippet code/src_corelib_io_qurl.cpp 10
325
326 If the two URLs were used via HTTP GET, the interpretation by the web
327 server would probably be different. In the first case, it would interpret
328 as one parameter, with a key of "q" and value "a+=b&c". In the second
329 case, it would probably interpret as two parameters, one with a key of "q"
330 and value "a =b", and the second with a key "c" and no value.
331
332 \sa QUrl::FormattingOptions
333*/

627// From RFC 3986, Appendix A Collected ABNF for URI
628// URI = scheme ":" hier-part [ "?" query ] [ "#" fragment ]
629//[...]
630// scheme = ALPHA *( ALPHA / DIGIT / "+" / "-" / "." )
631//
632// authority = [ userinfo "@" ] host [ ":" port ]
633// userinfo = *( unreserved / pct-encoded / sub-delims / ":" )
634// host = IP-literal / IPv4address / reg-name
635// port = *DIGIT
636//[...]
637// reg-name = *( unreserved / pct-encoded / sub-delims )
638//[..]
639// pchar = unreserved / pct-encoded / sub-delims / ":" / "@"
640//
641// query = *( pchar / "/" / "?" )
642//
643// fragment = *( pchar / "/" / "?" )
644//
645// pct-encoded = "%" HEXDIG HEXDIG
646//
647// unreserved = ALPHA / DIGIT / "-" / "." / "_" / "~"
648// reserved = gen-delims / sub-delims
649// gen-delims = ":" / "/" / "?" / "#" / "[" / "]" / "@"
650// sub-delims = "!" / "$" / "&" / "'" / "(" / ")"
651// / "*" / "+" / "," / ";" / "="
652// the path component has a complex ABNF that basically boils down to
653// slash-separated segments of "pchar"
654
655// The above is the strict definition of the URL components and we mostly
656// adhere to it, with few exceptions. QUrl obeys the following behavior:
657// - percent-encoding sequences always use uppercase HEXDIG;
658// - unreserved characters are *always* decoded, no exceptions;
659// - the space character and bytes with the high bit set are controlled by
660// the EncodeSpaces and EncodeUnicode bits;
661// - control characters, the percent sign itself, and bytes with the high
662// bit set that don't form valid UTF-8 sequences are always encoded,
663// except in FullyDecoded mode;
664// - sub-delims are always left alone, except in FullyDecoded mode;
665// - gen-delim change behavior depending on which section of the URL (or
666// the entire URL) we're looking at; see below;
667// - characters not mentioned above, like "<", and ">", are usually
668// decoded in individual sections of the URL, but encoded when the full
669// URL is put together (we can change on subjective definition of
670// "pretty").
671//
672// The behavior for the delimiters bears some explanation. The spec says in
673// section 2.2:
674// URIs that differ in the replacement of a reserved character with its
675// corresponding percent-encoded octet are not equivalent.
676// (note: QUrl API mistakenly uses the "reserved" term, so we will refer to
677// them here as "delimiters").
678//
679// For that reason, we cannot encode delimiters found in decoded form and we
680// cannot decode the ones found in encoded form if that would change the
681// interpretation. Conversely, we *can* perform the transformation if it would
682// not change the interpretation. From the last component of a URL to the first,
683// here are the gen-delims we can unambiguously transform when the field is
684// taken in isolation:
685// - fragment: none, since it's the last
686// - query: "#" is unambiguous
687// - path: "#" and "?" are unambiguous
688// - host: completely special but never ambiguous, see setHost() below.
689// - password: the "#", "?", "/", "[", "]" and "@" characters are unambiguous
690// - username: the "#", "?", "/", "[", "]", "@", and ":" characters are unambiguous
691// - scheme: doesn't accept any delimiter, see setScheme() below.
692//
693// Internally, QUrl stores each component in the format that corresponds to the
694// default mode (PrettyDecoded). It deviates from the "strict" FullyEncoded
695// mode in the following way:
696// - spaces are decoded
697// - valid UTF-8 sequences are decoded
698// - gen-delims that can be unambiguously transformed are decoded
699// - characters controlled by DecodeReserved are often decoded, though this behavior
700// can change depending on the subjective definition of "pretty"
701//
702// Note that the list of gen-delims that we can transform is different for the
703// user info (user name + password) and the authority (user info + host +
704// port).
705
706
707// list the recoding table modifications to be used with the recodeFromUser and
708// appendToUser functions, according to the rules above. Spaces and UTF-8
709// sequences are handled outside the tables.
710
711// the encodedXXX tables are run with the delimiters set to "leave" by default;
712// the decodedXXX tables are run with the delimiters set to "decode" by default
713// (except for the query, which doesn't use these functions)

718template <typename T> constexpr ushort encode(T x) noexcept { return ushort(0x200 | x); }

950inline bool QUrlPrivate::setScheme(const QString &value, qsizetype len, bool doSetError)
951{
952 // schemes are strictly RFC-compliant:
953 // scheme = ALPHA *( ALPHA / DIGIT / "+" / "-" / "." )
954 // we also lowercase the scheme
955
956 // schemes in URLs are not allowed to be empty, but they can be in
957 // "Relative URIs" which QUrl also supports. QUrl::setScheme does
958 // not call us with len == 0, so this can only be from parse()
959 scheme.clear();
960 if (len == 0)
961 return false;
962
963 sectionIsPresent |= Scheme;
964
965 // validate it:
966 qsizetype needsLowercasing = -1;
967 const ushort *p = reinterpret_cast<const ushort *>(value.data());
968 for (qsizetype i = 0; i < len; ++i) {
969 if (isAsciiLower(c: p[i]))
970 continue;
971 if (isAsciiUpper(c: p[i])) {
972 needsLowercasing = i;
973 continue;
974 }
975 if (i) {
976 if (isAsciiDigit(c: p[i]))
977 continue;
978 if (p[i] == '+' || p[i] == '-' || p[i] == '.')
979 continue;
980 }
981
982 // found something else
983 // don't call setError needlessly:
984 // if we've been called from parse(), it will try to recover
985 if (doSetError)
986 setError(errorCode: InvalidSchemeError, source: value, supplement: i);
987 return false;
988 }
989
990 scheme = value.left(n: len);
991
992 if (needsLowercasing != -1) {
993 // schemes are ASCII only, so we don't need the full Unicode toLower
994 QChar *schemeData = scheme.data(); // force detaching here
995 for (qsizetype i = needsLowercasing; i >= 0; --i) {
996 ushort c = schemeData[i].unicode();
997 if (isAsciiUpper(c))
998 schemeData[i] = QChar(c + 0x20);
999 }
1000 }
1001
1002 // did we set to the file protocol?
1003 if (scheme == fileScheme()
1004#ifdef Q_OS_WIN
1005 || scheme == webDavScheme()
1006#endif
1007 ) {
1008 flags |= IsLocalFile;
1009 } else {
1010 flags &= ~IsLocalFile;
1011 }
1012 return true;
1013}

1015inline void QUrlPrivate::setAuthority(const QString &auth, qsizetype from, qsizetype end, QUrl::ParsingMode mode)
1016{
1017 sectionIsPresent &= ~Authority;
1018 sectionIsPresent |= Host;
1019 port = -1;
1020
1021 // we never actually _loop_
1022 while (from != end) {
1023 qsizetype userInfoIndex = auth.indexOf(c: u'@', from);
1024 if (size_t(userInfoIndex) < size_t(end)) {
1025 setUserInfo(userInfo: auth, from, end: userInfoIndex);
1026 if (mode == QUrl::StrictMode && !validateComponent(section: UserInfo, input: auth, begin: from, end: userInfoIndex))
1027 break;
1028 from = userInfoIndex + 1;
1029 }
1030
1031 qsizetype colonIndex = auth.lastIndexOf(c: u':', from: end - 1);
1032 if (colonIndex < from)
1033 colonIndex = -1;
1034
1035 if (size_t(colonIndex) < size_t(end)) {
1036 if (auth.at(i: from).unicode() == '[') {
1037 // check if colonIndex isn't inside the "[...]" part
1038 qsizetype closingBracket = auth.indexOf(c: u']', from);
1039 if (size_t(closingBracket) > size_t(colonIndex))
1040 colonIndex = -1;
1041 }
1042 }
1043
1044 if (size_t(colonIndex) < size_t(end) - 1) {
1045 // found a colon with digits after it
1046 unsigned long x = 0;
1047 for (qsizetype i = colonIndex + 1; i < end; ++i) {
1048 ushort c = auth.at(i).unicode();
1049 if (isAsciiDigit(c)) {
1050 x *= 10;
1051 x += c - '0';
1052 } else {
1053 x = ulong(-1); // x != ushort(x)
1054 break;
1055 }
1056 }
1057 if (x == ushort(x)) {
1058 port = ushort(x);
1059 } else {
1060 setError(errorCode: InvalidPortError, source: auth, supplement: colonIndex + 1);
1061 if (mode == QUrl::StrictMode)
1062 break;
1063 }
1064 }
1065
1066 setHost(value: auth, from, end: qMin<size_t>(a: end, b: colonIndex), mode);
1067 if (mode == QUrl::StrictMode && !validateComponent(section: Host, input: auth, begin: from, end: qMin<size_t>(a: end, b: colonIndex))) {
1068 // clear host too
1069 sectionIsPresent &= ~Authority;
1070 break;
1071 }
1072
1073 // success
1074 return;
1075 }
1076 // clear all sections but host
1077 sectionIsPresent &= ~Authority | Host;
1078 userName.clear();
1079 password.clear();
1080 host.clear();
1081 port = -1;
1082}

1127// Host handling
1128// The RFC says the host is:
1129// host = IP-literal / IPv4address / reg-name
1130// IP-literal = "[" ( IPv6address / IPvFuture ) "]"
1131// IPvFuture = "v" 1*HEXDIG "." 1*( unreserved / sub-delims / ":" )
1132// [a strict definition of IPv6Address and IPv4Address]
1133// reg-name = *( unreserved / pct-encoded / sub-delims )
1134//
1135// We deviate from the standard in all but IPvFuture. For IPvFuture we accept
1136// and store only exactly what the RFC says we should. No percent-encoding is
1137// permitted in this field, so Unicode characters and space aren't either.
1138//
1139// For IPv4 addresses, we accept broken addresses like inet_aton does (that is,
1140// less than three dots). However, we correct the address to the proper form
1141// and store the corrected address. After correction, we comply to the RFC and
1142// it's exclusively composed of unreserved characters.
1143//
1144// For IPv6 addresses, we accept addresses including trailing (embedded) IPv4
1145// addresses, the so-called v4-compat and v4-mapped addresses. We also store
1146// those addresses like that in the hostname field, which violates the spec.
1147// IPv6 hosts are stored with the square brackets in the QString. It also
1148// requires no transformation in any way.
1149//
1150// As for registered names, it's the other way around: we accept only valid
1151// hostnames as specified by STD 3 and IDNA. That means everything we accept is
1152// valid in the RFC definition above, but there are many valid reg-names
1153// according to the RFC that we do not accept in the name of security. Since we
1154// do accept IDNA, reg-names are subject to ACE encoding and decoding, which is
1155// specified by the DecodeUnicode flag. The hostname is stored in its Unicode form.
1156
1157inline void QUrlPrivate::appendHost(QString &appendTo, QUrl::FormattingOptions options) const
1158{
1159 if (host.isEmpty())
1160 return;
1161 if (host.at(i: 0).unicode() == '[') {
1162 // IPv6 addresses might contain a zone-id which needs to be recoded
1163 if (options != 0)
1164 if (qt_urlRecode(appendTo, url: host, encoding: options, tableModifications: nullptr))
1165 return;
1166 appendTo += host;
1167 } else {
1168 // this is either an IPv4Address or a reg-name
1169 // if it is a reg-name, it is already stored in Unicode form
1170 if (options & QUrl::EncodeUnicode && !(options & 0x4000000))
1171 appendTo += qt_ACE_do(domain: host, op: ToAceOnly, dot: AllowLeadingDot, options: {});
1172 else
1173 appendTo += host;
1174 }
1175}

1177// the whole IPvFuture is passed and parsed here, including brackets;
1178// returns null if the parsing was successful, or the QChar of the first failure
1179static const QChar *parseIpFuture(QString &host, const QChar *begin, const QChar *end, QUrl::ParsingMode mode)
1180{
1181 // IPvFuture = "v" 1*HEXDIG "." 1*( unreserved / sub-delims / ":" )
1182 static const char acceptable[] =
1183 "!$&'()*+,;=" // sub-delims
1184 ":" // ":"
1185 "-._~"; // unreserved
1186
1187 // the brackets and the "v" have been checked
1188 const QChar *const origBegin = begin;
1189 if (begin[3].unicode() != '.')
1190 return &begin[3];
1191 if (isHexDigit(c: begin[2].unicode())) {
1192 // this is so unlikely that we'll just go down the slow path
1193 // decode the whole string, skipping the "[vH." and "]" which we already know to be there
1194 host += QStringView(begin, 4);
1195
1196 // uppercase the version, if necessary
1197 if (begin[2].unicode() >= 'a')
1198 host[host.size() - 2] = QChar{begin[2].unicode() - 0x20};
1199
1200 begin += 4;
1201 --end;
1202
1203 QString decoded;
1204 if (mode == QUrl::TolerantMode && qt_urlRecode(appendTo&: decoded, url: QStringView{begin, end}, encoding: QUrl::FullyDecoded, tableModifications: nullptr)) {
1205 begin = decoded.constBegin();
1206 end = decoded.constEnd();
1207 }
1208
1209 for ( ; begin != end; ++begin) {
1210 if (isAsciiLetterOrNumber(c: begin->unicode()))
1211 host += *begin;
1212 else if (begin->unicode() < 0x80 && strchr(s: acceptable, c: begin->unicode()) != nullptr)
1213 host += *begin;
1214 else
1215 return decoded.isEmpty() ? begin : &origBegin[2];
1216 }
1217 host += u']';
1218 return nullptr;
1219 }
1220 return &origBegin[2];
1221}

1224static const QChar *parseIp6(QString &host, const QChar *begin, const QChar *end, QUrl::ParsingMode mode)
1225{
1226 QStringView decoded(begin, end);
1227 QString decodedBuffer;
1228 if (mode == QUrl::TolerantMode) {
1229 // this struct is kept in automatic storage because it's only 4 bytes
1230 const ushort decodeColon[] = { decode(x: ':'), 0 };
1231 if (qt_urlRecode(appendTo&: decodedBuffer, url: decoded, encoding: QUrl::ComponentFormattingOption::PrettyDecoded, tableModifications: decodeColon))
1232 decoded = decodedBuffer;
1233 }
1234
1235 const QStringView zoneIdIdentifier(u"%25");
1236 QIPAddressUtils::IPv6Address address;
1237 QStringView zoneId;
1238
1239 qsizetype zoneIdPosition = decoded.indexOf(s: zoneIdIdentifier);
1240 if ((zoneIdPosition != -1) && (decoded.lastIndexOf(s: zoneIdIdentifier) == zoneIdPosition)) {
1241 zoneId = decoded.mid(pos: zoneIdPosition + zoneIdIdentifier.size());
1242 decoded.truncate(n: zoneIdPosition);
1243
1244 // was there anything after the zone ID separator?
1245 if (zoneId.isEmpty())
1246 return end;
1247 }
1248
1249 // did the address become empty after removing the zone ID?
1250 // (it might have always been empty)
1251 if (decoded.isEmpty())
1252 return end;
1253
1254 const QChar *ret = QIPAddressUtils::parseIp6(address, begin: decoded.constBegin(), end: decoded.constEnd());
1255 if (ret)
1256 return begin + (ret - decoded.constBegin());
1257
1258 host.reserve(asize: host.size() + (end - begin) + 2); // +2 for the brackets
1259 host += u'[';
1260 QIPAddressUtils::toString(appendTo&: host, address);
1261
1262 if (!zoneId.isEmpty()) {
1263 host += zoneIdIdentifier;
1264 host += zoneId;
1265 }
1266 host += u']';
1267 return nullptr;

1275
1276 const qsizetype len = end - begin;
1277 host.clear();
1278 sectionIsPresent |= Host;

1281
1282 if (begin[0].unicode() == '[') {

1284 // smallest IPv6 address is "[::]" (len = 4)
1285 // smallest IPvFuture address is "[v7.X]" (len = 6)
1286 if (end[-1].unicode() != ']') {
1287 setError(errorCode: HostMissingEndBracket, source: value);

1301 if (!c)

1305 setError(errorCode: InvalidIPv6AddressError, source: value, supplement: from);
1306 else
1307 setError(errorCode: InvalidCharacterInIPv6Error, source: value, supplement: c - value.constData());

1324 // percent encoding (e.g., "%31%30%2E%30%2E%30%2E%31" -> "10.0.0.1")

1362inline void QUrlPrivate::parse(const QString &url, QUrl::ParsingMode parsingMode)
1363{
1364 // URI-reference = URI / relative-ref
1365 // URI = scheme ":" hier-part [ "?" query ] [ "#" fragment ]
1366 // relative-ref = relative-part [ "?" query ] [ "#" fragment ]
1367 // hier-part = "//" authority path-abempty
1368 // / other path types
1369 // relative-part = "//" authority path-abempty
1370 // / other path types here
1371
1372 sectionIsPresent = 0;
1373 flags = 0;
1374 clearError();
1375
1376 // find the important delimiters
1377 qsizetype colon = -1;
1378 qsizetype question = -1;
1379 qsizetype hash = -1;
1380 const qsizetype len = url.size();
1381 const QChar *const begin = url.constData();
1382 const ushort *const data = reinterpret_cast<const ushort *>(begin);
1383
1384 for (qsizetype i = 0; i < len; ++i) {
1385 size_t uc = data[i];
1386 if (uc == '#' && hash == -1) {
1387 hash = i;
1388
1389 // nothing more to be found
1390 break;
1391 }
1392
1393 if (question == -1) {
1394 if (uc == ':' && colon == -1)
1395 colon = i;
1396 else if (uc == '?')
1397 question = i;
1398 }
1399 }
1400
1401 // check if we have a scheme
1402 qsizetype hierStart;
1403 if (colon != -1 && setScheme(value: url, len: colon, /* don't set error */ doSetError: false)) {
1404 hierStart = colon + 1;
1405 } else {
1406 // recover from a failed scheme: it might not have been a scheme at all
1407 scheme.clear();
1408 sectionIsPresent = 0;
1409 hierStart = 0;
1410 }
1411
1412 qsizetype pathStart;
1413 qsizetype hierEnd = qMin<size_t>(a: qMin<size_t>(a: question, b: hash), b: len);
1414 if (hierEnd - hierStart >= 2 && data[hierStart] == '/' && data[hierStart + 1] == '/') {
1415 // we have an authority, it ends at the first slash after these
1416 qsizetype authorityEnd = hierEnd;
1417 for (qsizetype i = hierStart + 2; i < authorityEnd ; ++i) {
1418 if (data[i] == '/') {
1419 authorityEnd = i;
1420 break;
1421 }
1422 }
1423
1424 setAuthority(auth: url, from: hierStart + 2, end: authorityEnd, mode: parsingMode);
1425
1426 // even if we failed to set the authority properly, let's try to recover
1427 pathStart = authorityEnd;
1428 setPath(value: url, from: pathStart, end: hierEnd);
1429 } else {
1430 userName.clear();
1431 password.clear();
1432 host.clear();
1433 port = -1;
1434 pathStart = hierStart;
1435
1436 if (hierStart < hierEnd)
1437 setPath(value: url, from: hierStart, end: hierEnd);
1438 else
1439 path.clear();
1440 }
1441
1442 if (size_t(question) < size_t(hash))
1443 setQuery(value: url, from: question + 1, iend: qMin<size_t>(a: hash, b: len));
1444
1445 if (hash != -1)
1446 setFragment(value: url, from: hash + 1, end: len);
1447
1448 if (error || parsingMode == QUrl::TolerantMode)
1449 return;
1450
1451 // The parsing so far was partially tolerant of errors, except for the
1452 // scheme parser (which is always strict) and the authority (which was
1453 // executed in strict mode).
1454 // If we haven't found any errors so far, continue the strict-mode parsing
1455 // from the path component onwards.
1456
1457 if (!validateComponent(section: Path, input: url, begin: pathStart, end: hierEnd))
1458 return;
1459 if (size_t(question) < size_t(hash) && !validateComponent(section: Query, input: url, begin: question + 1, end: qMin<size_t>(a: hash, b: len)))
1460 return;
1461 if (hash != -1)
1462 validateComponent(section: Fragment, input: url, begin: hash + 1, end: len);
1463}

1522/*
1523 From http://www.ietf.org/rfc/rfc3986.txt, 5.2.4: Remove dot segments
1524
1525 Removes unnecessary ../ and ./ from the path. Used for normalizing
1526 the URL.
1527*/
1528static void removeDotsFromPath(QString *path)
1529{
1530 // The input buffer is initialized with the now-appended path
1531 // components and the output buffer is initialized to the empty
1532 // string.
1533 QChar *out = path->data();
1534 const QChar *in = out;
1535 const QChar *end = out + path->size();
1536
1537 // If the input buffer consists only of
1538 // "." or "..", then remove that from the input
1539 // buffer;
1540 if (path->size() == 1 && in[0].unicode() == '.')
1541 ++in;
1542 else if (path->size() == 2 && in[0].unicode() == '.' && in[1].unicode() == '.')
1543 in += 2;
1544 // While the input buffer is not empty, loop:
1545 while (in < end) {
1546
1547 // otherwise, if the input buffer begins with a prefix of "../" or "./",
1548 // then remove that prefix from the input buffer;
1549 if (path->size() >= 2 && in[0].unicode() == '.' && in[1].unicode() == '/')
1550 in += 2;
1551 else if (path->size() >= 3 && in[0].unicode() == '.'
1552 && in[1].unicode() == '.' && in[2].unicode() == '/')
1553 in += 3;
1554
1555 // otherwise, if the input buffer begins with a prefix of
1556 // "/./" or "/.", where "." is a complete path segment,
1557 // then replace that prefix with "/" in the input buffer;
1558 if (in <= end - 3 && in[0].unicode() == '/' && in[1].unicode() == '.'
1559 && in[2].unicode() == '/') {
1560 in += 2;
1561 continue;
1562 } else if (in == end - 2 && in[0].unicode() == '/' && in[1].unicode() == '.') {
1563 *out++ = u'/';
1564 in += 2;
1565 break;
1566 }
1567
1568 // otherwise, if the input buffer begins with a prefix
1569 // of "/../" or "/..", where ".." is a complete path
1570 // segment, then replace that prefix with "/" in the
1571 // input buffer and remove the last //segment and its
1572 // preceding "/" (if any) from the output buffer;
1573 if (in <= end - 4 && in[0].unicode() == '/' && in[1].unicode() == '.'
1574 && in[2].unicode() == '.' && in[3].unicode() == '/') {
1575 while (out > path->constData() && (--out)->unicode() != '/')
1576 ;
1577 if (out == path->constData() && out->unicode() != '/')
1578 ++in;
1579 in += 3;
1580 continue;
1581 } else if (in == end - 3 && in[0].unicode() == '/' && in[1].unicode() == '.'
1582 && in[2].unicode() == '.') {
1583 while (out > path->constData() && (--out)->unicode() != '/')
1584 ;
1585 if (out->unicode() == '/')
1586 ++out;
1587 in += 3;
1588 break;
1589 }
1590
1591 // otherwise move the first path segment in
1592 // the input buffer to the end of the output
1593 // buffer, including the initial "/" character
1594 // (if any) and any subsequent characters up
1595 // to, but not including, the next "/"
1596 // character or the end of the input buffer.
1597 *out++ = *in++;
1598 while (in < end && in->unicode() != '/')
1599 *out++ = *in++;
1600 }
1601 path->truncate(pos: out - path->constData());
1602}

1604inline QUrlPrivate::ErrorCode QUrlPrivate::validityError(QString *source, qsizetype *position) const
1605{
1606 Q_ASSERT(!source == !position);
1607 if (error) {
1608 if (source) {
1609 *source = error->source;
1610 *position = error->position;
1611 }
1612 return error->code;
1613 }
1614
1615 // There are three more cases of invalid URLs that QUrl recognizes and they
1616 // are only possible with constructed URLs (setXXX methods), not with
1617 // parsing. Therefore, they are tested here.
1618 //
1619 // Two cases are a non-empty path that doesn't start with a slash and:
1620 // - with an authority
1621 // - without an authority, without scheme but the path with a colon before
1622 // the first slash
1623 // The third case is an empty authority and a non-empty path that starts
1624 // with "//".
1625 // Those cases are considered invalid because toString() would produce a URL
1626 // that wouldn't be parsed back to the same QUrl.
1627
1628 if (path.isEmpty())
1629 return NoError;
1630 if (path.at(i: 0) == u'/') {
1631 if (hasAuthority() || path.size() == 1 || path.at(i: 1) != u'/')
1632 return NoError;
1633 if (source) {
1634 *source = path;
1635 *position = 0;
1636 }
1637 return AuthorityAbsentAndPathIsDoubleSlash;
1638 }
1639

1641 if (source) {
1642 *source = path;
1643 *position = 0;
1644 }
1645 return AuthorityPresentAndPathIsRelative;
1646 }
1647 if (sectionIsPresent & QUrlPrivate::Scheme)

1652 ushort c = path.at(i).unicode();
1653 if (c == '/') {
1654 // found the slash before the colon
1655 return NoError;
1656 }
1657 if (c == ':') {
1658 // found the colon before the slash, it's invalid
1659 if (source) {
1660 *source = path;
1661 *position = i;
1662 }
1663 return RelativeUrlPathContainsColonBeforeSlash;
1664 }
1665 }

1669bool QUrlPrivate::validateComponent(QUrlPrivate::Section section, const QString &input,
1670 qsizetype begin, qsizetype end)
1671{
1672 // What we need to look out for, that the regular parser tolerates:
1673 // - percent signs not followed by two hex digits
1674 // - forbidden characters, which should always appear encoded
1675 // '"' / '<' / '>' / '\' / '^' / '`' / '{' / '|' / '}' / BKSP
1676 // control characters
1677 // - delimiters not allowed in certain positions
1678 // . scheme: parser is already strict
1679 // . user info: gen-delims except ":" disallowed ("/" / "?" / "#" / "[" / "]" / "@")
1680 // . host: parser is stricter than the standard
1681 // . port: parser is stricter than the standard
1682 // . path: all delimiters allowed
1683 // . fragment: all delimiters allowed
1684 // . query: all delimiters allowed
1685 static const char forbidden[] = "\"<>\\^`{|}\x7F";
1686 static const char forbiddenUserInfo[] = ":/?#[]@";
1687
1688 Q_ASSERT(section != Authority && section != Hierarchy && section != FullUrl);
1689
1690 const ushort *const data = reinterpret_cast<const ushort *>(input.constData());
1691 for (size_t i = size_t(begin); i < size_t(end); ++i) {
1692 uint uc = data[i];
1693 if (uc >= 0x80)
1694 continue;
1695
1696 bool error = false;
1697 if ((uc == '%' && (size_t(end) < i + 2 || !isHex(c: data[i + 1]) || !isHex(c: data[i + 2])))
1698 || uc <= 0x20 || strchr(s: forbidden, c: uc)) {
1699 // found an error
1700 error = true;
1701 } else if (section & UserInfo) {
1702 if (section == UserInfo && strchr(s: forbiddenUserInfo + 1, c: uc))
1703 error = true;
1704 else if (section != UserInfo && strchr(s: forbiddenUserInfo, c: uc))
1705 error = true;
1706 }
1707
1708 if (!error)
1709 continue;
1710
1711 ErrorCode errorCode = ErrorCode(int(section) << 8);
1712 if (section == UserInfo) {
1713 // is it the user name or the password?
1714 errorCode = InvalidUserNameError;
1715 for (size_t j = size_t(begin); j < i; ++j)
1716 if (data[j] == ':') {
1717 errorCode = InvalidPasswordError;
1718 break;
1719 }
1720 }
1721
1722 setError(errorCode, source: input, supplement: i);
1723 return false;
1724 }
1725
1726 // no errors
1727 return true;
1728}

3723 \list

3725 \li ftp.qt-project.org becomes ftp://ftp.qt-project.org
3726 \li hostname becomes http://hostname

3731 \a workingDirectory path. This is especially useful when handling command
3732 line arguments.

3736 as such if the file actually exists in the given working directory.

3738 flag AssumeLocalFile in \a options.
