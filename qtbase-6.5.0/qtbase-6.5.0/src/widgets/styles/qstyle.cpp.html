23static const int MaxBits = 8 * sizeof(QSizePolicy::ControlType);
24
25static int unpackControlTypes(QSizePolicy::ControlTypes controls, QSizePolicy::ControlType *array)
26{
27 if (!controls)
28 return 0;
29
30 // optimization: exactly one bit is set
31 if (qPopulationCount(v: uint(controls)) == 1) {
32 array[0] = QSizePolicy::ControlType(uint(controls));
33 return 1;
34 }
35
36 int count = 0;
37 for (int i = 0; i < MaxBits; ++i) {
38 if (uint bit = (controls & (0x1 << i)))
39 array[count++] = QSizePolicy::ControlType(bit);
40 }
41 return count;
42}

44/*!
45 \page qwidget-styling.html
46 \title Styling
47
48 Qt's built-in widgets use the QStyle class to perform nearly all
49 of their drawing. QStyle is an abstract base class that
50 encapsulates the look and feel of a GUI, and can be used to make
51 the widgets look exactly like the equivalent native widgets or to
52 give the widgets a custom look.
53
54 Qt provides a set of QStyle subclasses that emulate the native
55 look of the different platforms supported by Qt (QWindowsStyle,
56 QMacStyle, etc.). These styles are built into the
57 Qt GUI module, other styles can be made available using Qt's
58 plugin mechanism.
59
60 Most functions for drawing style elements take four arguments:
61
62 \list
63 \li an enum value specifying which graphical element to draw
64 \li a QStyleOption object specifying how and where to render that element
65 \li a QPainter object that should be used to draw the element
66 \li a QWidget object on which the drawing is performed (optional)
67 \endlist
68
69 The style gets all the information it needs to render the
70 graphical element from the QStyleOption class. The widget is
71 passed as the last argument in case the style needs it to perform
72 special effects (such as animated default buttons on \macos),
73 but it isn't mandatory. In fact, QStyle can be used to draw on any
74 paint device (not just widgets), in which case the widget argument
75 is a zero pointer.
76
77 \image paintsystem-stylepainter.png
78
79 The paint system also provides the QStylePainter class inheriting
80 from QPainter. QStylePainter is a convenience class for drawing
81 QStyle elements inside a widget, and extends QPainter with a set
82 of high-level drawing functions implemented on top of QStyle's
83 API. The advantage of using QStylePainter is that the parameter
84 lists get considerably shorter.
85
86 \table 100%
87 \row
88 \li \inlineimage paintsystem-icon.png
89 \li \b QIcon
90
91 The QIcon class provides scalable icons in different modes and states.
92
93 QIcon can generate pixmaps reflecting an icon's state, mode and
94 size. These pixmaps are generated from the set of pixmaps
95 made available to the icon, and are used by Qt widgets to show an
96 icon representing a particular action.
97
98 The rendering of a QIcon object is handled by the QIconEngine
99 class. Each icon has a corresponding icon engine that is
100 responsible for drawing the icon with a requested size, mode and
101 state.
102
103 \endtable

105 For more information about widget styling and appearance, see the
106 \l{Styles and Style Aware Widgets}.

197 \section1 Creating a Custom Style
198
199 You can create a custom look and feel for your application by
200 creating a custom style. There are two approaches to creating a
201 custom style. In the static approach, you either choose an
202 existing QStyle class, subclass it, and reimplement virtual
203 functions to provide the custom behavior, or you create an entire
204 QStyle class from scratch. In the dynamic approach, you modify the
205 behavior of your system style at runtime. The static approach is
206 described below. The dynamic approach is described in QProxyStyle.
207
208 The first step in the static approach is to pick one of the styles
209 provided by Qt from which you will build your custom style. Your
210 choice of QStyle class will depend on which style resembles your
211 desired style the most. The most general class that you can use as
212 a base is QCommonStyle (not QStyle). This is because Qt requires
213 its styles to be \l{QCommonStyle}s.
214
215 Depending on which parts of the base style you want to change,
216 you must reimplement the functions that are used to draw those
217 parts of the interface. To illustrate this, we will modify the
218 look of the spin box arrows drawn by QWindowsStyle. The arrows

290 \section1 Right-to-Left Desktops
291
292 Languages written from right to left (such as Arabic and Hebrew)
293 usually also mirror the whole layout of widgets, and require the
294 light to come from the screen's top-right corner instead of
295 top-left.
296
297 If you create a custom style, you should take special care when
298 drawing asymmetric elements to make sure that they also look
299 correct in a mirrored layout. An easy way to test your styles is
300 to run applications with the \c -reverse command-line option or
301 to call QApplication::setLayoutDirection() in your \c main()
302 function.
303
304 Here are some things to keep in mind when making a style work well in a
305 right-to-left environment:
306
307 \list
308 \li subControlRect() and subElementRect() return rectangles in screen coordinates
309 \li QStyleOption::direction indicates in which direction the item should be drawn in
310 \li If a style is not right-to-left aware it will display items as if it were left-to-right
311 \li visualRect(), visualPos(), and visualAlignment() are helpful functions that will
312 translate from logical to screen representations.
313 \li alignedRect() will return a logical rect aligned for the current direction
314 \endlist

316 \section1 Styles in Item Views
317
318 The painting of items in views is performed by a delegate. Qt's
319 default delegate, QStyledItemDelegate, is also used for calculating bounding
320 rectangles of items, and their sub-elements for the various kind
321 of item \l{Qt::ItemDataRole}{data roles}
322 QStyledItemDelegate supports. See the QStyledItemDelegate class
323 description to find out which datatypes and roles are supported. You
324 can read more about item data roles in \l{Model/View Programming}.
325
326 When QStyledItemDelegate paints its items, it draws
327 CE_ItemViewItem, and calculates their size with CT_ItemViewItem.
328 Note also that it uses SE_ItemViewItemText to set the size of
329 editors. When implementing a style to customize drawing of item
330 views, you need to check the implementation of QCommonStyle (and
331 any other subclasses from which your style
332 inherits). This way, you find out which and how
333 other style elements are painted, and you can then reimplement the
334 painting of elements that should be drawn differently.
335
336 We include a small example where we customize the drawing of item
337 backgrounds.
338
339 \snippet customviewstyle/customviewstyle.cpp 0
340
341 The primitive element PE_PanelItemViewItem is responsible for
342 painting the background of items, and is called from
343 \l{QCommonStyle}'s implementation of CE_ItemViewItem.
344
345 To add support for drawing of new datatypes and item data roles,
346 it is necessary to create a custom delegate. But if you only
347 need to support the datatypes implemented by the default
348 delegate, a custom style does not need an accompanying
349 delegate. The QStyledItemDelegate class description gives more
350 information on custom delegates.
351
352 The drawing of item view headers is also done by the style, giving
353 control over size of header items and row and column sizes.
354
355 \sa QStyleOption, QStylePainter, {Styles Example},

496 \fn QRect QStyle::itemTextRect(const QFontMetrics &metrics, const QRect &rectangle, int alignment, bool enabled, const QString &text) const
497
498 Returns the area within the given \a rectangle in which to draw
499 the provided \a text according to the specified font \a metrics
500 and \a alignment. The \a enabled parameter indicates whether or
501 not the associated item is enabled.
502
503 If the given \a rectangle is larger than the area needed to render
504 the \a text, the rectangle that is returned will be offset within
505 \a rectangle according to the specified \a alignment. For
506 example, if \a alignment is Qt::AlignCenter, the returned
507 rectangle will be centered within \a rectangle. If the given \a
508 rectangle is smaller than the area needed, the returned rectangle
509 will be the smallest rectangle large enough to render the \a text.
510
511 \sa Qt::Alignment
512*/
513QRect QStyle::itemTextRect(const QFontMetrics &metrics, const QRect &rect, int alignment, bool enabled,
514 const QString &text) const
515{
516 QRect result;
517 int x, y, w, h;
518 rect.getRect(ax: &x, ay: &y, aw: &w, ah: &h);
519 if (!text.isEmpty()) {
520 result = metrics.boundingRect(x, y, w, h, flags: alignment, text);
521 if (!enabled && proxy()->styleHint(stylehint: SH_EtchDisabledText)) {
522 result.setWidth(result.width()+1);
523 result.setHeight(result.height()+1);
524 }
525 } else {
526 result = QRect(x, y, w, h);
527 }
528 return result;
529}

531/*!
532 \fn QRect QStyle::itemPixmapRect(const QRect &rectangle, int alignment, const QPixmap &pixmap) const
533
534 Returns the area within the given \a rectangle in which to draw
535 the specified \a pixmap according to the defined \a alignment.
536*/
537QRect QStyle::itemPixmapRect(const QRect &rect, int alignment, const QPixmap &pixmap) const
538{
539 QRect result;
540 int x, y, w, h;
541 rect.getRect(ax: &x, ay: &y, aw: &w, ah: &h);
542
543 QSizeF pixmapSize = pixmap.deviceIndependentSize();
544 const int pixmapWidth = pixmapSize.width();
545 const int pixmapHeight = pixmapSize.height();
546
547 if ((alignment & Qt::AlignVCenter) == Qt::AlignVCenter)
548 y += h/2 - pixmapHeight/2;
549 else if ((alignment & Qt::AlignBottom) == Qt::AlignBottom)
550 y += h - pixmapHeight;
551 if ((alignment & Qt::AlignRight) == Qt::AlignRight)
552 x += w - pixmapWidth;
553 else if ((alignment & Qt::AlignHCenter) == Qt::AlignHCenter)
554 x += w/2 - pixmapWidth/2;
555 else if ((alignment & Qt::AlignLeft) != Qt::AlignLeft && QGuiApplication::isRightToLeft())
556 x += w - pixmapWidth;
557 result = QRect(x, y, pixmapWidth, pixmapHeight);
558 return result;
559}

562 \fn void QStyle::drawItemText(QPainter *painter, const QRect &rectangle, int alignment, const QPalette &palette, bool enabled, const QString& text, QPalette::ColorRole textRole) const
563
564 Draws the given \a text in the specified \a rectangle using the
565 provided \a painter and \a palette.
566
567 The text is drawn using the painter's pen, and aligned and wrapped
568 according to the specified \a alignment. If an explicit \a
569 textRole is specified, the text is drawn using the \a palette's
570 color for the given role. The \a enabled parameter indicates
571 whether or not the item is enabled; when reimplementing this
572 function, the \a enabled parameter should influence how the item is
573 drawn.
574
575 \sa Qt::Alignment, drawItemPixmap()
576*/
577void QStyle::drawItemText(QPainter *painter, const QRect &rect, int alignment, const QPalette &pal,
578 bool enabled, const QString& text, QPalette::ColorRole textRole) const
579{
580 if (text.isEmpty())
581 return;
582 QPen savedPen;
583 if (textRole != QPalette::NoRole) {
584 savedPen = painter->pen();
585 painter->setPen(QPen(pal.brush(cr: textRole), savedPen.widthF()));
586 }
587 if (!enabled) {
588 if (proxy()->styleHint(stylehint: SH_DitherDisabledText)) {
589 QRect br;
590 painter->drawText(r: rect, flags: alignment, text, br: &br);
591 painter->fillRect(br, QBrush(painter->background().color(), Qt::Dense5Pattern));
592 return;
593 } else if (proxy()->styleHint(stylehint: SH_EtchDisabledText)) {
594 QPen pen = painter->pen();
595 painter->setPen(pal.light().color());
596 painter->drawText(r: rect.adjusted(xp1: 1, yp1: 1, xp2: 1, yp2: 1), flags: alignment, text);
597 painter->setPen(pen);
598 }
599 }
600 painter->drawText(r: rect, flags: alignment, text);
601 if (textRole != QPalette::NoRole)
602 painter->setPen(savedPen);
603}

2213 parameter maps to 0, \a max maps to \a span and other values are

2222
2223 \sa sliderValueFromPosition()
2224*/

2226int QStyle::sliderPositionFromValue(int min, int max, int logicalValue, int span, bool upsideDown)
2227{
2228 if (span <= 0 || max <= min)
2229 return 0;
2230 if (logicalValue < min)
2231 return upsideDown ? span : 0;
2232 if (logicalValue > max)
2233 return upsideDown ? 0 : span;

2235 const uint range = qint64(max) - min;
2236 const uint p = upsideDown ? qint64(max) - logicalValue : qint64(logicalValue) - min;
2237
2238 if (range > (uint)INT_MAX/4096) {
2239 double dpos = (double(p))/(double(range)/span);
2240 return int(dpos);
2241 } else if (range > (uint)span) {
2242 return (2 * p * span + range) / (2*range);
2243 } else {
2244 uint div = span / range;
2245 uint mod = span % range;
2246 return p * div + (2 * p * mod + range) / (2 * range);
2247 }
2248 // equiv. to (p * span) / range + 0.5
2249 // no overflow because of this implicit assumption:
2250 // span <= 4096

2272{

2275 if (pos >= span)
2276 return upsideDown ? min : max;
2277
2278 const qint64 range = qint64(max) - min;
