4/*!
5 \page qtconcurrentrun.html
6 \title Concurrent Run
7 \brief A simple way to run a task in a separate thread.
8 \ingroup thread
9
10 The QtConcurrent::run() function runs a function in a separate thread.
11 The return value of the function is made available through the QFuture API.
12
13 QtConcurrent::run() is an overloaded method. You can think of these overloads as slightly
14 different \e modes.
15 In \l {Concurrent Run (basic mode)} {basic mode}, the function passed to QtConcurrent::run()
16 is able to report merely a single computation result to its caller.
17 In \l {Concurrent Run With Promise} {run with promise mode}, the function passed to
18 QtConcurrent::run() can make use of the additional
19 QPromise API, which enables multiple result reporting, progress reporting,
20 suspending the computation when requested by the caller, or stopping
21 the computation on the caller's demand.
22
23 This function is a part of the Qt Concurrent framework.
24
25 \section1 Concurrent Run (basic mode)
26
27 The function passed to QtConcurrent::run() may report the result
28 through its return value.
29

124 \section1 Concurrent Run With Promise
125
126 The \e {Run With Promise} mode enables more control for the running
127 task compared to \e basic mode of QtConcurrent::run().
128 It allows progress reporting of the running task,
129 reporting multiple results, suspending the execution
130 if it was requested, or canceling the task on caller's
131 demand.
132
133 \section2 The mandatory QPromise argument
134
135 The function passed to QtConcurrent::run() in \e {Run With Promise} mode
136 is expected to have an additional argument of \c {QPromise<T> &} type, where
137 \c T is the type of the computation result (it should match the type \c T
138 of QFuture<T> returned by QtConcurrent::run()), like e.g.:
139
140 \snippet code/src_concurrent_qtconcurrentrun.cpp 9
141
142 The \c promise argument is instantiated inside the QtConcurrent::run()
143 function, and its reference is passed to the invoked \c aFunction, so the
144 user doesn't need to instantiate it, nor pass it explicitly
145 when calling QtConcurrent::run() in this mode.
146
147 The additional argument of QPromise type always needs to appear
148 as a first argument on function's arguments list, like:
149
150 \snippet code/src_concurrent_qtconcurrentrun.cpp 10
151
152 \section2 Reporting results
153
154 In contrast to \e basic mode of QtConcurrent::run(), the function passed to
155 QtConcurrent::run() in \e {Run With Promise} mode is expected to always return void type.
156 Result reporting is done through the additional argument of QPromise type.
157 It also enables multiple result reporting, like:
158
159 \snippet code/src_concurrent_qtconcurrentrun.cpp 11
160
161 \note There's no need to call QPromise::start() and QPromise::finish() to
162 indicate the beginning and the end of computation (like you would normally do when
163 using QPromise). QtConcurrent::run() will always call them before starting and
164 after finishing the execution.
165
166 \section2 Suspending and canceling the execution
167
168 The QPromise API also enables suspending and canceling the computation, if requested:
169
170 \snippet code/src_concurrent_qtconcurrentrun.cpp 12
171
172 The call to \c future.suspend() requests the running task to
173 hold its execution. After calling this method, the running task
174 will suspend after the next call to \c promise.suspendIfRequested()
175 in its iteration loop. In this case the running task will
176 block on a call to \c promise.suspendIfRequested(). The blocked
177 call will unblock after the \c future.resume() is called.
178 Note, that internally suspendIfRequested() uses wait condition
179 in order to unblock, so the running thread goes into an idle state
180 instead of wasting its resources when blocked in order to periodically
181 check if the resume request came from the caller's thread.
182
183 The call to \c future.cancel() from the last line causes that the next
184 call to \c promise.isCanceled() will return \c true and
185 \c aFunction will return immediately without any further result reporting.
186
187 \note There's no need to call QPromise::finish() to stop the computation
188 after the cancellation (like you would normally do when using QPromise).
189 QtConcurrent::run() will always call it after finishing the execution.
190
191 \section2 Progress reporting
192
193 It's also possible to report the progress of a task
194 independently of result reporting, like:
195
196 \snippet code/src_concurrent_qtconcurrentrun.cpp 13
197
198 The caller installs the \c QFutureWatcher for the \c QFuture
199 returned by QtConcurrent::run() in order to
200 connect to its \c progressValueChanged() signal and update

222/*!
223 \fn QFuture<T> QtConcurrent::run(Function function, ...);
224
225 Equivalent to
226 \code
227 QtConcurrent::run(QThreadPool::globalInstance(), function, ...);
228 \endcode
229
230 Runs \a function in a separate thread. The thread is taken from the global
231 QThreadPool. Note that \a function may not run immediately; \a function
232 will only be run once a thread becomes available.
233
234//! [run-description]
235 In \l {Concurrent Run (basic mode)} {basic mode} T is the same type as the return value
236 of \a function. Non-void return values can be accessed via the QFuture::result() function.
237
238 In \l {Concurrent Run (basic mode)} {basic mode} the QFuture returned can only be used to
239 query for the running/finished status and the return value of the function. In particular,
240 canceling or pausing can be issued only if the computations behind the future
241 has not been started.
242
243 In \l {Concurrent Run With Promise} {run with promise mode}, the \a function is expected
244 to return void and must take an additional argument of \c {QPromise<T> &} type,
245 placed as a first argument in function's argument list. T is the result type
246 and it is the same for the returned \c QFuture<T>.
247
248 In \l {Concurrent Run With Promise} {run with promise mode}, similar to \e basic mode, the
249 QFuture returned can be used to query for the running/finished status and the value reported
250 by the function. In addition, it may be used for suspending or canceling the
251 running task, fetching multiple results from the called \a function or
252 monitoring progress reported by the \a function.
253
254 \sa {Concurrent Run (basic mode)}, {Concurrent Run With Promise}
255//! [run-description]
256*/
257
258/*!
259 \since 5.4
260 \fn QFuture<T> QtConcurrent::run(QThreadPool *pool, Function function, ...);
261
262 Runs \a function in a separate thread. The thread is taken from the
263 QThreadPool \a pool. Note that \a function may not run immediately; \a function
264 will only be run once a thread becomes available.
265
266 \include qtconcurrentrun.cpp run-description
267*/
