13/*!
14 \class QShaderDescription
15 \internal
16 \inmodule QtGui
17
18 \brief Describes the interface of a shader.
19
20 A shader typically has a set of inputs and outputs. A vertex shader for
21 example has a number of input variables and may use one or more uniform
22 buffers to access data (e.g. a modelview matrix) provided by the
23 application. The shader for the fragment stage receives data from the
24 vertex stage (in a simple setup) and may also rely on data from uniform
25 buffers, images, and samplers.
26
27 When it comes to vertex inputs and the layout of the uniform buffers (what
28 are the names of the members? what is there size, offset, and so on),
29 applications and frameworks may need to discover this dynamically at run
30 time. This is typical when the shader is not built-in but provided by an
31 external entity, like the user.
32
33 Modern and lean graphics APIs may no longer provide a way to query shader
34 reflection information at run time. Therefore, such data is now
35 automatically generated by QShaderBaker and is provided as a
36 QShaderDescription object for each and every QShader.
37
38 \section2 Example
39
40 Take the following vertex shader:
41
42 \badcode
43 #version 440
44
45 layout(location = 0) in vec4 position;
46 layout(location = 1) in vec3 color;
47 layout(location = 0) out vec3 v_color;
48
49 layout(std140, binding = 0) uniform buf {
50 mat4 mvp;
51 float opacity;
52 } ubuf;
53
54 out gl_PerVertex { vec4 gl_Position; };
55
56 void main()
57 {
58 v_color = color;
59 gl_Position = ubuf.mvp * position;
60 }
61 \endcode
62
63 This shader has two inputs: \c position at location 0 with a type of \c
64 vec4, and \c color at location 1 with a type of \c vec3. It has one output:
65 \c v_color, although this is typically not interesting for applications.
66 What is more important, there is a uniform block at binding 0 with a size
67 of 68 bytes and two members, a 4x4 matrix named \c mvp at offset 0, and a
68 float \c opacity at offset 64.
69
70 All this is described by a QShaderDescription object. QShaderDescription can
71 be serialized to JSON and to a binary format via QDataStream, and can be
72 deserialized from this binary format. In practice this is rarely needed
73 since QShader takes care of the associated QShaderDescription automatically,
74 but if the QShaderDescription of the above shader would be written out as
75 JSON (like it is done by the \c qsb tool's \c{-d} option), it would look
76 like the following:
77
78 \badcode
79 {
80 "inputs": [
81 {
82 "location": 1,
83 "name": "color",
84 "type": "vec3"
85 },
86 {
87 "location": 0,
88 "name": "position",
89 "type": "vec4"
90 }
91 ],
92 "outputs": [
93 {
94 "location": 0,
95 "name": "v_color",
96 "type": "vec3"
97 }
98 ],
99 "uniformBlocks": [
100 {
101 "binding": 0,
102 "blockName": "buf",
103 "members": [
104 {
105 "matrixStride": 16,
106 "name": "mvp",
107 "offset": 0,
108 "size": 64,
109 "type": "mat4"
110 },
111 {
112 "name": "opacity",
113 "offset": 64,
114 "size": 4,
115 "type": "float"
116 }
117 ],
118 "set": 0,
119 "size": 68,
120 "structName": "ubuf"
121 }
122 ]
123 }
124 \endcode
125
126 The C++ API allows accessing a data structure like the above. For
127 simplicity the inner structs only contain public data members, also
128 considering that their layout is unlikely to change in the future.
129
130 \sa QShaderBaker, QShader
131 */

296/*!
297 \return true if the QShaderDescription contains at least one entry in one of
298 the variable and block lists.
299 */
300bool QShaderDescription::isValid() const
301{
302 return !d->inVars.isEmpty() || !d->outVars.isEmpty()
303 || !d->uniformBlocks.isEmpty() || !d->pushConstantBlocks.isEmpty() || !d->storageBlocks.isEmpty()
304 || !d->combinedImageSamplers.isEmpty() || !d->storageImages.isEmpty()
305 || !d->separateImages.isEmpty() || !d->separateSamplers.isEmpty()
306 || !d->inBuiltins.isEmpty() || !d->outBuiltins.isEmpty();
307}

383/*!
384 \return the list of shader storage blocks.
385
386 For example, with GLSL/Vulkan shaders as source, the declaration
387
388 \badcode
389 struct Stuff {
390 vec2 a;
391 vec2 b;
392 };
393 layout(std140, binding = 0) buffer StuffSsbo {
394 vec4 whatever;
395 Stuff stuff[];
396 } buf;
397 \endcode
398
399 generates the following: (shown as textual JSON here)
400
401 \badcode
402 "storageBlocks": [ {
403 "binding": 0,
404 "blockName": "StuffSsbo",
405 "instanceName": "buf",
406 "knownSize": 16,
407 "runtimeArrayStride": 16
408 "members": [
409 {
410 "name": "whatever",
411 "offset": 0,
412 "size": 16,
413 "type": "vec4"
414 },
415 {
416 "arrayDims": [
417 0
418 ],
419 "name": "stuff",
420 "offset": 16,
421 "size": 0,
422 "structMembers": [
423 {
424 "name": "a",
425 "offset": 0,
426 "size": 8,
427 "type": "vec2"
428 },
429 {
430 "name": "b",
431 "offset": 8,
432 "size": 8,
433 "type": "vec2"
434 }
435 ],
436 "type": "struct"
437 }
438 ],
439 "set": 0
440 } ]
441 \endcode
442
443 \note The size of the last member in the storage block is undefined. This shows
444 up as \c size 0 and an array dimension of \c{[0]}. The storage block's \c knownSize
445 excludes the size of the last member since that will only be known at run time. The
446 stride in bytes between array items for a last member with undefined array size is
447 \c runtimeArrayStride. This value is determined according to the specified buffer
448 memory layout standard (std140, std430) rules.
449
450 \note SSBOs are not available with some graphics APIs, such as, OpenGL 2.x or
451 OpenGL ES older than 3.1.
452 */
453QList<QShaderDescription::StorageBlock> QShaderDescription::storageBlocks() const
454{
455 return d->storageBlocks;
456}
