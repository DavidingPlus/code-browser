31/*!
32 \class QOpenGLWidget
33 \inmodule QtOpenGLWidgets
34 \since 5.4
35
36 \brief The QOpenGLWidget class is a widget for rendering OpenGL graphics.
37
38 QOpenGLWidget provides functionality for displaying OpenGL graphics
39 integrated into a Qt application. It is very simple to use: Make
40 your class inherit from it and use the subclass like any other
41 QWidget, except that you have the choice between using QPainter and
42 standard OpenGL rendering commands.
43
44 QOpenGLWidget provides three convenient virtual functions that you
45 can reimplement in your subclass to perform the typical OpenGL
46 tasks:
47
48 \list
49 \li paintGL() - Renders the OpenGL scene. Gets called whenever the widget
50 needs to be updated.
51 \li resizeGL() - Sets up the OpenGL viewport, projection, etc. Gets
52 called whenever the widget has been resized (and also when it
53 is shown for the first time because all newly created widgets get a
54 resize event automatically).
55 \li initializeGL() - Sets up the OpenGL resources and state. Gets called
56 once before the first time resizeGL() or paintGL() is called.
57 \endlist
58
59 If you need to trigger a repaint from places other than paintGL() (a
60 typical example is when using \l{QTimer}{timers} to animate scenes),
61 you should call the widget's update() function to schedule an update.
62
63 Your widget's OpenGL rendering context is made current when
64 paintGL(), resizeGL(), or initializeGL() is called. If you need to
65 call the standard OpenGL API functions from other places (e.g. in
66 your widget's constructor or in your own paint functions), you
67 must call makeCurrent() first.
68
69 All rendering happens into an OpenGL framebuffer
70 object. makeCurrent() ensure that it is bound in the context. Keep
71 this in mind when creating and binding additional framebuffer
72 objects in the rendering code in paintGL(). Never re-bind the
73 framebuffer with ID 0. Instead, call defaultFramebufferObject() to
74 get the ID that should be bound.
75
76 QOpenGLWidget allows using different OpenGL versions and profiles
77 when the platform supports it. Just set the requested format via
78 setFormat(). Keep in mind however that having multiple QOpenGLWidget
79 instances in the same window requires that they all use the same
80 format, or at least formats that do not make the contexts
81 non-sharable. To overcome this issue, prefer using
82 QSurfaceFormat::setDefaultFormat() instead of setFormat().
83
84 \note Calling QSurfaceFormat::setDefaultFormat() before constructing
85 the QApplication instance is mandatory on some platforms (for example,
86 \macos) when an OpenGL core profile context is requested. This is to
87 ensure that resource sharing between contexts stays functional as all
88 internal contexts are created using the correct version and profile.
89
90 \section1 Painting Techniques
91
92 As described above, subclass QOpenGLWidget to render pure 3D content in the
93 following way:
94
95 \list
96
97 \li Reimplement the initializeGL() and resizeGL() functions to
98 set up the OpenGL state and provide a perspective transformation.
99
100 \li Reimplement paintGL() to paint the 3D scene, calling only
101 OpenGL functions.
102
103 \endlist
104
105 It is also possible to draw 2D graphics onto a QOpenGLWidget subclass using QPainter:
106
107 \list
108
109 \li In paintGL(), instead of issuing OpenGL commands, construct a QPainter
110 object for use on the widget.
111
112 \li Draw primitives using QPainter's member functions.
113
114 \li Direct OpenGL commands can still be issued. However, you must make sure
115 these are enclosed by a call to the painter's beginNativePainting() and
116 endNativePainting().
117
118 \endlist
119
120 When performing drawing using QPainter only, it is also possible to perform
121 the painting like it is done for ordinary widgets: by reimplementing paintEvent().
122
123 \list
124
125 \li Reimplement the paintEvent() function.
126
127 \li Construct a QPainter object targeting the widget. Either pass the widget to the
128 constructor or the QPainter::begin() function.
129
130 \li Draw primitives using QPainter's member functions.
131
132 \li Painting finishes then the QPainter instance is destroyed. Alternatively,
133 call QPainter::end() explicitly.
134
135 \endlist

137 \section1 OpenGL Function Calls, Headers and QOpenGLFunctions
138
139 When making OpenGL function calls, it is strongly recommended to avoid calling
140 the functions directly. Instead, prefer using QOpenGLFunctions (when making
141 portable applications) or the versioned variants (for example,
142 QOpenGLFunctions_3_2_Core and similar, when targeting modern, desktop-only
143 OpenGL). This way the application will work correctly in all Qt build
144 configurations, including the ones that perform dynamic OpenGL implementation
145 loading which means applications are not directly linking to an GL
146 implementation and thus direct function calls are not feasible.
147
148 In paintGL() the current context is always accessible by calling
149 QOpenGLContext::currentContext(). From this context an already initialized,
150 ready-to-be-used QOpenGLFunctions instance is retrievable by calling
151 QOpenGLContext::functions(). An alternative to prefixing every GL call is to
152 inherit from QOpenGLFunctions and call
153 QOpenGLFunctions::initializeOpenGLFunctions() in initializeGL().
154
155 As for the OpenGL headers, note that in most cases there will be no need to
156 directly include any headers like GL.h. The OpenGL-related Qt headers will
157 include qopengl.h which will in turn include an appropriate header for the
158 system. This might be an OpenGL ES 3.x or 2.0 header, the highest version that
159 is available, or a system-provided gl.h. In addition, a copy of the extension
160 headers (called glext.h on some systems) is provided as part of Qt both for
161 OpenGL and OpenGL ES. These will get included automatically on platforms where
162 feasible. This means that constants and function pointer typedefs from ARB,
163 EXT, OES extensions are automatically available.

193 \section1 Multisampling
194
195 To enable multisampling, set the number of requested samples on the
196 QSurfaceFormat that is passed to setFormat(). On systems that do not support
197 it the request may get ignored.
198
199 Multisampling support requires support for multisampled renderbuffers and
200 framebuffer blits. On OpenGL ES 2.0 implementations it is likely that these
201 will not be present. This means that multisampling will not be available. With
202 modern OpenGL versions and OpenGL ES 3.0 and up this is usually not a problem
203 anymore.
204
205 \section1 Threading
206
207 Performing offscreen rendering on worker threads, for example to generate
208 textures that are then used in the GUI/main thread in paintGL(), are supported
209 by exposing the widget's QOpenGLContext so that additional contexts sharing
210 with it can be created on each thread.
211
212 Drawing directly to the QOpenGLWidget's framebuffer outside the GUI/main
213 thread is possible by reimplementing paintEvent() to do nothing. The context's
214 thread affinity has to be changed via QObject::moveToThread(). After that,
215 makeCurrent() and doneCurrent() are usable on the worker thread. Be careful to
216 move the context back to the GUI/main thread afterwards.
217
218 Triggering a buffer swap just for the QOpenGLWidget is not possible since there
219 is no real, onscreen native surface for it. It is up to the widget stack to
220 manage composition and buffer swaps on the gui thread. When a thread is done
221 updating the framebuffer, call update() \b{on the GUI/main thread} to
222 schedule composition.
223
224 Extra care has to be taken to avoid using the framebuffer when the GUI/main
225 thread is performing compositing. The signals aboutToCompose() and
226 frameSwapped() will be emitted when the composition is starting and
227 ending. They are emitted on the GUI/main thread. This means that by using a
228 direct connection aboutToCompose() can block the GUI/main thread until the
229 worker thread has finished its rendering. After that, the worker thread must
230 perform no further rendering until the frameSwapped() signal is emitted. If
231 this is not acceptable, the worker thread has to implement a double buffering
232 mechanism. This involves drawing using an alternative render target, that is
233 fully controlled by the thread, e.g. an additional framebuffer object, and
234 blitting to the QOpenGLWidget's framebuffer at a suitable time.

236 \section1 Context Sharing
237
238 When multiple QOpenGLWidgets are added as children to the same top-level
239 widget, their contexts will share with each other. This does not apply for
240 QOpenGLWidget instances that belong to different windows.
241
242 This means that all QOpenGLWidgets in the same window can access each other's
243 sharable resources, like textures, and there is no need for an extra "global
244 share" context.
245
246 To set up sharing between QOpenGLWidget instances belonging to different
247 windows, set the Qt::AA_ShareOpenGLContexts application attribute before
248 instantiating QApplication. This will trigger sharing between all
249 QOpenGLWidget instances without any further steps.
250
251 Creating extra QOpenGLContext instances that share resources like textures
252 with the QOpenGLWidget's context is also possible. Simply pass the pointer
253 returned from context() to QOpenGLContext::setShareContext() before calling
254 QOpenGLContext::create(). The resulting context can also be used on a
255 different thread, allowing threaded generation of textures and asynchronous
256 texture uploads.
257
258 Note that QOpenGLWidget expects a standard conformant implementation of
259 resource sharing when it comes to the underlying graphics drivers. For
260 example, some drivers, in particular for mobile and embedded hardware, have
261 issues with setting up sharing between an existing context and others that are
262 created later. Some other drivers may behave in unexpected ways when trying to
263 utilize shared resources between different threads.
264
265 \section1 Resource Initialization and Cleanup
266
267 The QOpenGLWidget's associated OpenGL context is guaranteed to be current
268 whenever initializeGL() and paintGL() are invoked. Do not attempt to create
269 OpenGL resources before initializeGL() is called. For example, attempting to
270 compile shaders, initialize vertex buffer objects or upload texture data will
271 fail when done in a subclass's constructor. These operations must be deferred
272 to initializeGL(). Some of Qt's OpenGL helper classes, like QOpenGLBuffer or
273 QOpenGLVertexArrayObject, have a matching deferred behavior: they can be
274 instantiated without a context, but all initialization is deferred until a
275 create(), or similar, call. This means that they can be used as normal
276 (non-pointer) member variables in a QOpenGLWidget subclass, but the create()
277 or similar function can only be called from initializeGL(). Be aware however
278 that not all classes are designed like this. When in doubt, make the member
279 variable a pointer and create and destroy the instance dynamically in
280 initializeGL() and the destructor, respectively.
281
282 Releasing the resources also needs the context to be current. Therefore
283 destructors that perform such cleanup are expected to call makeCurrent()
284 before moving on to destroy any OpenGL resources or wrappers. Avoid deferred
285 deletion via \l{QObject::deleteLater()}{deleteLater()} or the parenting
286 mechanism of QObject. There is no guarantee the correct context will be
287 current at the time the instance in question is really destroyed.
288

330 the application. Therefore the safest and most robust is always to perform
331 explicit cleanup for all resources and resource wrappers used in the
332 QOpenGLWidget.

334 \section1 Limitations and Other Considerations
335
336 Putting other widgets underneath and making the QOpenGLWidget transparent will
337 not lead to the expected results: The widgets underneath will not be
338 visible. This is because in practice the QOpenGLWidget is drawn before all
339 other regular, non-OpenGL widgets, and so see-through type of solutions are
340 not feasible. Other type of layouts, like having widgets on top of the
341 QOpenGLWidget, will function as expected.
342
343 When absolutely necessary, this limitation can be overcome by setting the
344 Qt::WA_AlwaysStackOnTop attribute on the QOpenGLWidget. Be aware however that
345 this breaks stacking order, for example it will not be possible to have other
346 widgets on top of the QOpenGLWidget, so it should only be used in situations
347 where a semi-transparent QOpenGLWidget with other widgets visible underneath
348 is required.
349
350 Note that this does not apply when there are no other widgets underneath and
351 the intention is to have a semi-transparent window. In that case the
352 traditional approach of setting Qt::WA_TranslucentBackground
353 on the top-level window is sufficient. Note that if the transparent areas are
354 only desired in the QOpenGLWidget, then Qt::WA_NoSystemBackground will need
355 to be turned back to \c false after enabling Qt::WA_TranslucentBackground.
356 Additionally, requesting an alpha channel for the QOpenGLWidget's context via
357 setFormat() may be necessary too, depending on the system.
358
359 QOpenGLWidget supports multiple update behaviors, just like QOpenGLWindow. In
360 preserved mode the rendered content from the previous paintGL() call is
361 available in the next one, allowing incremental rendering. In non-preserved
362 mode the content is lost and paintGL() implementations are expected to redraw
363 everything in the view.
364
365 Before Qt 5.5 the default behavior of QOpenGLWidget was to preserve the
366 rendered contents between paintGL() calls. Since Qt 5.5 the default behavior
367 is non-preserved because this provides better performance and the majority of
368 applications have no need for the previous content. This also resembles the
369 semantics of an OpenGL-based QWindow and matches the default behavior of
370 QOpenGLWindow in that the color and ancillary buffers are invalidated for
371 each frame. To restore the preserved behavior, call setUpdateBehavior() with
372 \c PartialUpdate.
373
374 \note When dynamically adding a QOpenGLWidget into a widget hierarchy, e.g.
375 by parenting a new QOpenGLWidget to a widget where the corresponding
376 top-level widget is already shown on screen, the associated native window may
377 get implicitly destroyed and recreated if the QOpenGLWidget is the first of
378 its kind within its window. This is because the window type changes from
379 \l{QSurface::RasterSurface}{RasterSurface} to
380 \l{QSurface::OpenGLSurface}{OpenGLSurface} and that has platform-specific
381 implications. This behavior is new in Qt 6.4.
382
383 Once a QOpenGLWidget is added to a widget hierarchy, the contents of the
384 top-level window is flushed via OpenGL-based rendering. Widgets other than
385 the QOpenGLWidget continue to draw their content using a software-based
386 painter, but the final composition is done through the 3D API.
387
388 \note Displaying a QOpenGLWidget requires an alpha channel in the associated
389 top-level window's backing store due to the way composition with other
390 QWidget-based content works. If there is no alpha channel, the content
391 rendered by the QOpenGLWidget will not be visible. This can become
392 particularly relevant on Linux/X11 in remote display setups (such as, with
393 Xvnc), when using a color depth lower than 24. For example, a color depth of
394 16 will typically map to using a backing store image with the format
395 QImage::Format_RGB16 (RGB565), leaving no room for an alpha
396 channel. Therefore, if experiencing problems with getting the contents of a
397 QOpenGLWidget composited correctly with other the widgets in the window, make
398 sure the server (such as, vncserver) is configured with a 24 or 32 bit depth
399 instead of 16.
400
401 \section1 Alternatives
402
403 Adding a QOpenGLWidget into a window turns on OpenGL-based
404 compositing for the entire window. In some special cases this may
405 not be ideal, and the old QGLWidget-style behavior with a separate,
406 native child window is desired. Desktop applications that understand
407 the limitations of this approach (for example when it comes to
408 overlaps, transparency, scroll views and MDI areas), can use
409 QOpenGLWindow with QWidget::createWindowContainer(). This is a
410 modern alternative to QGLWidget and is faster than QOpenGLWidget due
411 to the lack of the additional composition step. It is strongly
412 recommended to limit the usage of this approach to cases where there
413 is no other choice. Note that this option is not suitable for most
414 embedded and mobile platforms, and it is known to have issues on
415 certain desktop platforms (e.g. \macos) too. The stable,
416 cross-platform solution is always QOpenGLWidget.
417

419 \section1 Stereoscopic rendering
420
421 Starting from 6.5 QOpenGLWidget has support for stereoscopic rendering.
422 To enable it, set the QSurfaceFormat::StereoBuffers flag
423 globally before the window is created, using QSurfaceFormat::SetDefaultFormat().
424
425 \note Using setFormat() will not necessarily work because of how the flag is
426 handled internally.
427
428 This will trigger paintGL() to be called twice each frame,
429 once for each QOpenGLWidget::TargetBuffer. In paintGL(), call
430 currentTargetBuffer() to query which one is currently being drawn to.
431
432 \note For more control over the left and right color buffers, consider using
433 QOpenGLWindow + QWidget::createWindowContainer() instead.
434
435 \note This type of 3D rendering has certain hardware requirements,
436 like the graphics card needs to be setup with stereo support.
437

589void QOpenGLWidgetPaintDevicePrivate::beginPaint()
590{
591 // NB! autoFillBackground is and must be false by default. Otherwise we would clear on
592 // every QPainter begin() which is not desirable. This is only for legacy use cases,
593 // like using QOpenGLWidget as the viewport of a graphics view, that expect clearing
594 // with the palette's background color.
595 if (w->autoFillBackground()) {
596 QOpenGLFunctions *f = QOpenGLContext::currentContext()->functions();
597 if (w->format().hasAlpha()) {
598 f->glClearColor(red: 0, green: 0, blue: 0, alpha: 0);
599 } else {
600 QColor c = w->palette().brush(cr: w->backgroundRole()).color();
601 float alpha = c.alphaF();
602 f->glClearColor(red: c.redF() * alpha, green: c.greenF() * alpha, blue: c.blueF() * alpha, alpha);
603 }
604 f->glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT | GL_STENCIL_BUFFER_BIT);
605 }
606}

717void QOpenGLWidgetPrivate::recreateFbos()
718{
719 Q_Q(QOpenGLWidget);
720
721 emit q->aboutToResize();
722
723 context->makeCurrent(surface);
724
725 destroyFbos();
726
727 int samples = requestedSamples;
728 QOpenGLExtensions *extfuncs = static_cast<QOpenGLExtensions *>(context->functions());
729 if (!extfuncs->hasOpenGLExtension(extension: QOpenGLExtensions::FramebufferMultisample))
730 samples = 0;
731
732 QOpenGLFramebufferObjectFormat format;
733 format.setAttachment(QOpenGLFramebufferObject::CombinedDepthStencil);
734 format.setSamples(samples);
735 if (textureFormat)
736 format.setInternalTextureFormat(textureFormat);
737
738 const QSize deviceSize = q->size() * q->devicePixelRatio();
739 fbos[QOpenGLWidget::LeftBuffer] = new QOpenGLFramebufferObject(deviceSize, format);
740 if (samples > 0)
741 resolvedFbos[QOpenGLWidget::LeftBuffer] = new QOpenGLFramebufferObject(deviceSize);
742
743 const bool stereo = isStereoEnabled();
744
745 if (stereo) {
746 fbos[QOpenGLWidget::RightBuffer] = new QOpenGLFramebufferObject(deviceSize, format);
747 if (samples > 0)
748 resolvedFbos[QOpenGLWidget::RightBuffer] = new QOpenGLFramebufferObject(deviceSize);
749 }
750
751 textureFormat = fbos[QOpenGLWidget::LeftBuffer]->format().internalTextureFormat();
752
753 currentTargetBuffer = QOpenGLWidget::LeftBuffer;
754 fbos[currentTargetBuffer]->bind();
755 context->functions()->glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT | GL_STENCIL_BUFFER_BIT);
756 ensureRhiDependentResources();
757
758 if (stereo) {
759 currentTargetBuffer = QOpenGLWidget::RightBuffer;
760 fbos[currentTargetBuffer]->bind();
761 context->functions()->glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT | GL_STENCIL_BUFFER_BIT);
762 ensureRhiDependentResources();
763 currentTargetBuffer = QOpenGLWidget::LeftBuffer;
764 }
765
766 flushPending = true; // Make sure the FBO is initialized before use
767
768 paintDevice->setSize(deviceSize);
769 paintDevice->setDevicePixelRatio(q->devicePixelRatio());
770
771 emit q->resized();
772}

817void QOpenGLWidgetPrivate::initialize()
818{
819 Q_Q(QOpenGLWidget);
820 if (initialized)
821 return;
822
823 // If no global shared context get our toplevel's context with which we
824 // will share in order to make the texture usable by the underlying window's backingstore.
825 QWidget *tlw = q->window();
826 QWidgetPrivate *tlwd = get(w: tlw);
827
828 // Do not include the sample count. Requesting a multisampled context is not necessary
829 // since we render into an FBO, never to an actual surface. What's more, attempting to
830 // create a pbuffer with a multisampled config crashes certain implementations. Just
831 // avoid the entire hassle, the result is the same.
832 requestedSamples = requestedFormat.samples();
833 requestedFormat.setSamples(0);
834
835 QRhi *rhi = nullptr;
836 if (QWidgetRepaintManager *repaintManager = tlwd->maybeRepaintManager())
837 rhi = repaintManager->rhi();
838
839 // Could be that something else already initialized the window with some
840 // other graphics API for the QRhi, that's not good.
841 if (rhi && rhi->backend() != QRhi::OpenGLES2) {
842 qWarning(msg: "The top-level window is not using OpenGL for composition, '%s' is not compatible with QOpenGLWidget",
843 rhi->backendName());
844 return;
845 }
846
847 // If rhi or contextFromRhi is null, showing content on-screen will not work.
848 // However, offscreen rendering and grabFramebuffer() will stay fully functional.
849
850 QOpenGLContext *contextFromRhi = rhi ? static_cast<const QRhiGles2NativeHandles *>(rhi->nativeHandles())->context : nullptr;
851
852 context = new QOpenGLContext;
853 context->setFormat(requestedFormat);
854 if (contextFromRhi) {
855 context->setShareContext(contextFromRhi);
856 context->setScreen(contextFromRhi->screen());
857 }
858 if (Q_UNLIKELY(!context->create())) {
859 qWarning(msg: "QOpenGLWidget: Failed to create context");
860 return;
861 }
862
863 surface = new QOffscreenSurface;
864 surface->setFormat(context->format());
865 surface->setScreen(context->screen());
866 surface->create();
867
868 if (Q_UNLIKELY(!context->makeCurrent(surface))) {
869 qWarning(msg: "QOpenGLWidget: Failed to make context current");
870 return;
871 }
872
873 // Propagate settings that make sense only for the tlw. Note that this only
874 // makes sense for properties that get picked up even after the native
875 // window is created.
876 if (tlw->windowHandle()) {
877 QSurfaceFormat tlwFormat = tlw->windowHandle()->format();
878 if (requestedFormat.swapInterval() != tlwFormat.swapInterval()) {
879 // Most platforms will pick up the changed swap interval on the next
880 // makeCurrent or swapBuffers.
881 tlwFormat.setSwapInterval(requestedFormat.swapInterval());
882 tlw->windowHandle()->setFormat(tlwFormat);
883 }
884 if (requestedFormat.swapBehavior() != tlwFormat.swapBehavior()) {
885 tlwFormat.setSwapBehavior(requestedFormat.swapBehavior());
886 tlw->windowHandle()->setFormat(tlwFormat);
887 }
888 }
889
890 paintDevice = new QOpenGLWidgetPaintDevice(q);
891 paintDevice->setSize(q->size() * q->devicePixelRatio());
892 paintDevice->setDevicePixelRatio(q->devicePixelRatio());
893
894 initialized = true;
895
896 q->initializeGL();
897}

916void QOpenGLWidgetPrivate::render()
917{
918 Q_Q(QOpenGLWidget);
919
920 if (fakeHidden || !initialized)
921 return;
922
923 setCurrentTargetBuffer(QOpenGLWidget::LeftBuffer);
924
925 QOpenGLContext *ctx = QOpenGLContext::currentContext();
926 if (!ctx) {
927 qWarning(msg: "QOpenGLWidget: No current context, cannot render");
928 return;
929 }
930
931 if (!fbos[QOpenGLWidget::LeftBuffer]) {
932 qWarning(msg: "QOpenGLWidget: No fbo, cannot render");
933 return;
934 }
935
936 const bool stereo = isStereoEnabled();
937 if (stereo) {
938 static bool warningGiven = false;
939 if (!fbos[QOpenGLWidget::RightBuffer] && !warningGiven) {
940 qWarning(msg: "QOpenGLWidget: Stereo is enabled, but no right buffer. Using only left buffer");
941 warningGiven = true;
942 }
943 }
944
945 if (updateBehavior == QOpenGLWidget::NoPartialUpdate && hasBeenComposed) {
946 invalidateFbo();
947
948 if (stereo && fbos[QOpenGLWidget::RightBuffer]) {
949 setCurrentTargetBuffer(QOpenGLWidget::RightBuffer);
950 invalidateFbo();
951 setCurrentTargetBuffer(QOpenGLWidget::LeftBuffer);
952 }
953
954 hasBeenComposed = false;
955 }
956
957 QOpenGLFunctions *f = ctx->functions();
958 f->glViewport(x: 0, y: 0, width: q->width() * q->devicePixelRatio(), height: q->height() * q->devicePixelRatio());
959 inPaintGL = true;
960
961 QOpenGLContextPrivate::get(context: ctx)->defaultFboRedirect = fbos[currentTargetBuffer]->handle();
962 q->paintGL();
963
964 if (stereo && fbos[QOpenGLWidget::RightBuffer]) {
965 setCurrentTargetBuffer(QOpenGLWidget::RightBuffer);
966 QOpenGLContextPrivate::get(context: ctx)->defaultFboRedirect = fbos[currentTargetBuffer]->handle();
967 q->paintGL();
968 }
969 QOpenGLContextPrivate::get(context: ctx)->defaultFboRedirect = 0;
970
971 inPaintGL = false;
972 flushPending = true;
973}

975void QOpenGLWidgetPrivate::invalidateFbo()
976{
977 QOpenGLExtensions *f = static_cast<QOpenGLExtensions *>(QOpenGLContext::currentContext()->functions());
978 if (f->hasOpenGLExtension(extension: QOpenGLExtensions::DiscardFramebuffer)) {
979 const int gl_color_attachment0 = 0x8CE0; // GL_COLOR_ATTACHMENT0
980 const int gl_depth_attachment = 0x8D00; // GL_DEPTH_ATTACHMENT
981 const int gl_stencil_attachment = 0x8D20; // GL_STENCIL_ATTACHMENT
982#ifdef Q_OS_WASM
983 // webgl does not allow separate depth and stencil attachments
984 // QTBUG-69913
985 const int gl_depth_stencil_attachment = 0x821A; // GL_DEPTH_STENCIL_ATTACHMENT
986
987 const GLenum attachments[] = {
988 gl_color_attachment0, gl_depth_attachment, gl_stencil_attachment, gl_depth_stencil_attachment
989 };
990#else
991 const GLenum attachments[] = {
992 gl_color_attachment0, gl_depth_attachment, gl_stencil_attachment
993 };
994#endif
995 f->glDiscardFramebufferEXT(GL_FRAMEBUFFER, numAttachments: sizeof attachments / sizeof *attachments, attachments);
996 } else {
997 f->glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT | GL_STENCIL_BUFFER_BIT);
998 }

1019QImage QOpenGLWidgetPrivate::grabFramebuffer(QOpenGLWidget::TargetBuffer targetBuffer)
1020{
1021 Q_Q(QOpenGLWidget);
1022
1023 initialize();
1024 if (!initialized)
1025 return QImage();
1026
1027 // The second fbo is only created when stereoscopic rendering is enabled
1028 // Just use the default one if not.
1029 if (targetBuffer == QOpenGLWidget::RightBuffer && !isStereoEnabled())
1030 targetBuffer = QOpenGLWidget::LeftBuffer;
1031
1032 if (!fbos[targetBuffer]) // could be completely offscreen, without ever getting a resize event
1033 recreateFbos();
1034
1035 if (!inPaintGL)
1036 render();
1037
1038 setCurrentTargetBuffer(targetBuffer);
1039 if (resolvedFbos[targetBuffer]) {
1040 resolveSamplesForBuffer(targetBuffer);
1041 resolvedFbos[targetBuffer]->bind();
1042 }
1043

1046 res.setDevicePixelRatio(q->devicePixelRatio());
1047
1048 // While we give no guarantees of what is going to be left bound, prefer the
1049 // multisample fbo instead of the resolved one. Clients may continue to
1050 // render straight after calling this function.
1051 if (resolvedFbos[targetBuffer]) {
1052 setCurrentTargetBuffer(targetBuffer);
1053 }
1054
1055 return res;

1540 \reimp
1541*/
1542int QOpenGLWidget::metric(QPaintDevice::PaintDeviceMetric metric) const
1543{
1544 Q_D(const QOpenGLWidget);
1545 if (d->inBackingStorePaint)
1546 return QWidget::metric(metric);
1547
1548 auto window = d->windowHandle(mode: QWidgetPrivate::WindowHandleMode::TopLevel);
1549 QScreen *screen = window ? window->screen() : QGuiApplication::primaryScreen();
1550
1551 const float dpmx = qt_defaultDpiX() * 100. / 2.54;
1552 const float dpmy = qt_defaultDpiY() * 100. / 2.54;
1553
1554 switch (metric) {
1555 case PdmWidth:
1556 return width();
1557 case PdmHeight:
1558 return height();
1559 case PdmDepth:
1560 return 32;
1561 case PdmWidthMM:
1562 if (screen)
1563 return width() * screen->physicalSize().width() / screen->geometry().width();
1564 else
1565 return width() * 1000 / dpmx;
1566 case PdmHeightMM:
1567 if (screen)
1568 return height() * screen->physicalSize().height() / screen->geometry().height();
1569 else
1570 return height() * 1000 / dpmy;
1571 case PdmNumColors:
1572 return 0;
1573 case PdmDpiX:
1574 if (screen)
1575 return qRound(d: screen->logicalDotsPerInchX());
1576 else
1577 return qRound(d: dpmx * 0.0254);
1578 case PdmDpiY:
1579 if (screen)
1580 return qRound(d: screen->logicalDotsPerInchY());
1581 else
1582 return qRound(d: dpmy * 0.0254);
1583 case PdmPhysicalDpiX:
1584 if (screen)
1585 return qRound(d: screen->physicalDotsPerInchX());
1586 else
1587 return qRound(d: dpmx * 0.0254);
1588 case PdmPhysicalDpiY:
1589 if (screen)
1590 return qRound(d: screen->physicalDotsPerInchY());
1591 else
1592 return qRound(d: dpmy * 0.0254);
1593 case PdmDevicePixelRatio:
1594 if (window)
1595 return int(window->devicePixelRatio());
1596 else
1597 return 1.0;
1598 case PdmDevicePixelRatioScaled:
1599 if (window)
1600 return int(window->devicePixelRatio() * devicePixelRatioFScale());
1601 else
1602 return int(devicePixelRatioFScale());
1603 default:
1604 qWarning(msg: "QOpenGLWidget::metric(): unknown metric %d", metric);
1605 return 0;
1606 }

1654 \reimp
1655*/
1656bool QOpenGLWidget::event(QEvent *e)
1657{
1658 Q_D(QOpenGLWidget);
1659 switch (e->type()) {
1660 case QEvent::WindowAboutToChangeInternal:
1661 d->resetRhiDependentResources();
1662 break;
1663 case QEvent::WindowChangeInternal:
1664 if (QCoreApplication::testAttribute(attribute: Qt::AA_ShareOpenGLContexts))
1665 break;
1666 if (d->initialized)
1667 d->reset();
1668 if (isHidden())
1669 break;
1670 Q_FALLTHROUGH();
1671 case QEvent::Show: // reparenting may not lead to a resize so reinitialize on Show too
1672 if (d->initialized && !d->wrapperTextures[d->currentTargetBuffer] && window()->windowHandle()) {
1673 // Special case: did grabFramebuffer() for a hidden widget that then became visible.
1674 // Recreate all resources since the context now needs to share with the TLW's.
1675 if (!QCoreApplication::testAttribute(attribute: Qt::AA_ShareOpenGLContexts))
1676 d->reset();
1677 }
1678 if (QWidgetRepaintManager *repaintManager = QWidgetPrivate::get(w: window())->maybeRepaintManager()) {
1679 if (!d->initialized && !size().isEmpty() && repaintManager->rhi()) {
1680 d->initialize();
1681 if (d->initialized) {
1682 d->recreateFbos();
1683 // QTBUG-89812: generate a paint event, like resize would do,
1684 // otherwise a QOpenGLWidget in a QDockWidget may not show the
1685 // content upon (un)docking.
1686 d->sendPaintEvent(toBePainted: QRect(QPoint(0, 0), size()));
1687 }
1688 }
1689 }
1690 break;
1691 case QEvent::ScreenChangeInternal:
1692 if (d->initialized && d->paintDevice->devicePixelRatio() != devicePixelRatio())
1693 d->recreateFbos();
1694 break;
1695 default:
1696 break;
1697 }
1698 return QWidget::event(event: e);
1699}
